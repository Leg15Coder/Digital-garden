==Опр.== **Thread Pool (Пул потоков)** - объект, создающий фиксированное количество рабочих потоков, которые живут всё время работы приложения и повторно используются для выполнения поступающих задач.

> [!tip] Оптимизация
> Thread pool решает [[Жизненный цикл TCP-сервера#Проблема C10k|проблему C10k]], ускоряя обработку данных.

### Полная реализация Thread Pool через [[Condvar|условную переменную]]

```c++
#include <vector>
#include <queue>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <functional>

class ThreadPool {
private:
    std::vector<std::thread> workers;          // Пул созданных потоков
    std::queue<std::functional<void()>> tasks; // Очередь задач
    
    std::mutex queue_mutex;                    // Мьютекс для защиты очереди задач
    std::condition_variable cv;                // Сигнал для засыпания/пробуждения работников
    bool stop = false;                         // Флаг остановки пула

public:
    ThreadPool(size_t threads) {
        for (size_t i = 0; i < threads; ++i) {
            workers.emplace_back([this] {
                while (true) {
                    std::functional<void()> task;
                    {
                        std::unique_lock<std::mutex> lock(this->queue_mutex);
                        // Поток спит, пока не появится задача или сигнал стоп
                        this->cv.wait(lock, [this] { 
                            return this->stop || !this->tasks.empty(); 
                        });
                        
                        // Если пул закрывается и задач не осталось - завершаем поток
                        if (this->stop && this->tasks.empty()) {
                            return;
                        }
                        
                        // Забираем задачу из очереди
                        task = std::move(this->tasks.front());
                        this->tasks.pop();
                    } 
                    // Выполняем задачу ВНЕ лока мьютекса!
                    task(); 
                }
            });
        }
    }

    // Добавление новой задачи в пул
    void enqueue(std::functional<void()> task) {
        {
            std::unique_lock<std::mutex> lock(queue_mutex);
            tasks.push(task);
        }
        cv.notify_one(); // Будим один свободный поток
    }

    ~ThreadPool() {
        {
            std::unique_lock<std::mutex> lock(queue_mutex);
            stop = true;
        }
        cv.notify_all(); // Пробуждаем все потоки для корректного завершения
        for (std::thread &worker : workers) {
            if (worker.joinable()) {
                worker.join();
            }
        }
    }
};
```

### Фундаментальная уязвимость наивного Thread Pool

- Thread Pool всё ещё уязвим при использовании **блокирующего ввода-вывода**.

#### Сценарий «Атаки N клиентов»:

Представьте, что в запущенном TCP-сервере выделено в Thread Pool ровно N потоков. Каждое входящее соединение передаёт функцию `handle_client` в пул задач. Внутри этой функции происходит стандартное блокирующее чтение из сети: `read(client_socket, buf, size);`

Если злоумышленник *(или N медленных клиентов)* откроет одновременно N соединений и **не отправит ни одного байта данных**, произойдёт катастрофа:

1. Все N потоков пула возьмут эти N задач.
2. Каждый поток зайдёт в системный вызов `read()`.
3. Поскольку данных нет, а вызов блокирующий, операционная система усыпит эти потоки до момента появления данных.
4. **Результат:** Пул потоков полностью парализован *(все работники заняты вечным ожиданием)*. Когда к серверу подключится (N+1)-й легитимный клиент, его задача встанет в очередь, но её абсолютно некому будет обработать. **Сервер уходит в отказ в обслуживании**.

> [!warning] Вывод
> Thread Pool эффективен для вычислительных задач (CPU-bound) или неблокирующего I/O. Использование блокирующих сетевых вызовов внутри фиксированного пула потоков делает систему уязвимой к deadlock ресурсов.

### Важные моменты

1. Почему в реализации Thread Pool вызов `task()` происходит строго после выхода из блока мьютекса?
    - Если запустить `task()` под заблокированным мьютексом очереди, то на всё время выполнения этой задачи очередь будет заблокирована. Другие потоки пула не смогут забрать свои задачи из очереди, и параллельное выполнение превратится в строго последовательное (однопоточное).
