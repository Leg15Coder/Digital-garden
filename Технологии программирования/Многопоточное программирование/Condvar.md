Представьте сценарий, когда одному потоку нужно дождаться выполнения некоторого условия, которое должен изменить другой поток (например, дождаться, пока в очереди появится задача для обработки). Как такое можно реализовать?

> [!fail] Неэффективный подход (Busy Waiting)
> Поток крутится в бесконечном цикле проверки флага: `while (!ready) {}`. При этом ядро процессора загружается на 100%, выполняя абсолютно бесполезную работу по постоянному опросу переменной.

> [!done] Правильно:
> Использовать условную переменную (condvar).

==Опр.== **Условная переменная (condvar)** - это механизм синхронизации, позволяющий потоку заблокироваться (заснуть, полностью освободив процессор) до тех пор, пока другой поток не изменит состояние данных и не оповестит об этом.

### Как это устроено в C++?

Для корректной работы условной переменной всегда требуются три компонента:
1. Сама условная переменная: `std::condition_variable cv;`
2. Мьютекс, защищающий разделяемое состояние: `std::mutex mtx;`
3. Переменная состояния (предикат), отражающая условие (например, `bool ready = false;` или размер очереди `tasks.empty()`).

#### Ключевые методы:

- `wait((std::unique_lock<std::mutex>& lock))`: усыпляет текущий поток. При вызове `wait` поток _атомарно отпускает мьютекс_ и засыпает. Это нужно, чтобы другие потоки могли захватить мьютекс и изменить условие. Как только поток пробуждается, он _автоматически заново захватывает мьютекс_ перед выходом из функции `wait`.
- `notify_one()`: будит ровно один из потоков, ожидающих на данной условной переменной.
- `notify_all()`: будит абсолютно все потоки, которые в данный момент заблокированы на этой условной переменной.

### Ложные пробуждения (Spurious Wakeups)

==Опр.== **Ложное пробуждение** - это ситуация, когда поток выходит из состояния ожидания сам по себе, даже если никто не вызывал `notify_one()` или `notify_all()`, либо если условие на самом деле еще не выполнено.

#### Почему это происходит?

Это обусловлено архитектурой ядер современных операционных систем *(включая Linux и Windows)*. Гарантия полного отсутствия ложных пробуждений на уровне ОС потребовала бы колоссального усложнения кода ядра и сильно замедлила бы работу примитивов синхронизации. Стандарты API *(POSIX threads и C++)* осознанно допускают ложные пробуждения, перекладывая обязанность повторной проверки условия на программиста.

#### Пример улучшения кода (Борьба с ложным пробуждением)

> [!bug] Плохой код:
```c++
std::unique_lock<std::mutex> lock(mtx);
if (!ready) {} // Использование условия if вместо цикла while!
// ОШИБКА: Если произойдет ложное пробуждение, код пойдет дальше, UB.
```

> [!done] Хороший код:
```c++
std::unique_lock<std::mutex> lock(mtx);
while (!ready) { // Всегда проверяем условие в цикле while!
    cv.wait(lock); // При ложном пробуждении поток вернется к проверке ready
}
// Гарантировано: мы пройдём сюда только если ready == true.
```

> [!tip] Синтаксический сахар
> В C++ есть переопределённый метод `wait`, автоматически оборачивающий ожидание в цикл `while`:
```c++
cv.wait(lock, [] { return ready; }); // Передача лямбда-предиката
```

### Реализация [[Mutex#Семафоры `std counting_semaphore` (C++20)|семафора]] через Convar

```c++
class CorrectSemaphore {
private:
    std::mutex mtx;
    std::condition_variable cv;
    int count;
public:
    explicit CorrectSemaphore(int init_count) : count(init_count) {}

    void acquire() {
        std::unique_lock<std::mutex> lock(mtx);
        // Защищаемся от ложных пробуждений: спим до тех пор, пока ресурс равен 0
        while (count == 0) {
            cv.wait(lock);
        }
        count--;         // Забираем один ресурс
    }

    void release() {
        std::unique_lock<std::mutex> lock(mtx);
        count++;         // Освобождаем ресурс
        cv.notify_one(); // Оповещаем один ждущий поток
    }
};
```

### Важные моменты

1. Почему в метод `cv.wait()` мы обязаны передавать именно `std::unique_lock`, а не более лёгкий `std::lock_guard`?
    - `std::lock_guard` не поддерживает ручной вызов методов блокировки и разблокировки. `std::condition_variable::wait()` устроен так, что ему необходимо самостоятельно отпускать мьютекс при засыпании и захватывать его обратно при пробуждении. На это способен только `std::unique_lock`.

2. Когда использовать `notify_one()`, а когда `notify_all()`?
    - `notify_one()` используют, когда все ждущие потоки выполняют эквивалентные задачи и только один из них сможет обработать изменение состояния. `notify_all()` используется, когда потоки ждут разных условий или когда логика требует пробуждения всех *(например, при завершении работы пула или в барьерах синхронизации)*.
