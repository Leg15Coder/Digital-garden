В однопоточном программировании действует правило **As-if**: компилятор и процессор имеют право производить любые оптимизации (переставлять инструкции местами, удалять «ненужные» обращения к памяти, кэшировать переменные в регистрах), если это не меняет наблюдаемое поведение программы с точки зрения этого единственного потока.

В многопоточной среде оптимизации компилятора и аппаратные особенности процессора (такие как внеочередное выполнение инструкций `Out-of-Order Execution` и наличие буферов записи `Store Buffers`) приводят к тому, что потоки могут видеть операции с памятью в совершенно разном порядке.

==Опр.== **Модель памяти C++** *(введена в C++11)* - это формальный контракт между разработчиком, компилятором и аппаратным обеспечением. Она строго определяет, при каких условиях изменения памяти, сделанные одним потоком, гарантированно становятся видимыми для другого потока.

### Data Race

Основополагающее правило модели памяти: если два потока одновременно обращаются к одной и той же ячейке памяти, причём хотя бы один из них производит запись, и эти обращения **не синхронизированы**, в программе возникает **Data Race**.
- В C++ любая гонка по данным является **UB**. Компилятор вправе полностью разрушить логику программы при наличии UB.
- Единственный способ избежать Data Race - использование примитивов синхронизации (mutex) или атомарных типов данных (`std::atomic`\).

## Формальные отношения порядка

Для строгого доказательства корректности многопоточных алгоритмов модель памяти оперирует абстрактными математическими отношениями частичного порядка между вычислениями.

### А. Sequenced-before

Это отношение строгого частичного порядка между операциями в рамках **одного потока**.
- Если в коде выражение `A` расположено до выражения `B` и правила языка гарантируют последовательное выполнение (например, точка с запятой `;` или операторы `&&`, `||`, `,`), то `A` является _sequenced-before_ `B`.
- Это чисто внутрипоточная абстракция.

### Б. Synchronizes-with

Это отношение между операциями в **разных потоках**. Оно создаётся примитивами синхронизации или операциями над atomics.
- Атомарная операция записи `A` в потоке `1` над переменной `M` с семантикой _Release_ (или _Seq_cst_) **synchronizes-with** атомарной операцией чтения `B` в потоке `2` над той же переменной `M` с семантикой _Acquire_ (или _Seq_cst_), если операция `B` считала значение, которое было записано операцией `A`.
- Также `unlock()` на мьютексе _synchronizes-with_ последующим успешным `lock()` на том же мьютексе в другом потоке.

### В. Happens-before

Главное отношение видимости памяти. Если операция `A happens-before B`, то результат операции `A` гарантированно виден операции `B`. Формально оно определяется транзитивным замыканием:
1. Если `A` является _sequenced-before_ `B`, то `A happens-before B`.
2. Если `A` _synchronizes-with_ `B`, то `A inter-thread happens-before B`.
3. Если `A happens-before X` и `X happens-before B`, то `A happens-before B`.

> **Связь с Data Race**: Если две операции над одной ячейкой памяти (одна из которых запись) не связаны отношением `happens-before`, они образуют Data Race.

## Разновидности (std::memory_order)

По умолчанию все операции над `std::atomic` используют самый строгий порядок `std::memory_order_seq_cst`. Однако C++ позволяет явно указывать флаги упорядочивания для оптимизации производительности под конкретные архитектуры процессоров.

```C++
#include <atomic>

std::atomic<int> x{0};
// Явное указание порядка памяти при записи и чтении
x.store(42, std::memory_order_release);
int val = x.load(std::memory_order_acquire);
```

### 1. `std::memory_order_relaxed`

- **Гарантии**: Только атомарность самой операции *(переменная не будет прочитана частично)* и когерентность кэша *(все потоки видят изменения одной конкретной переменной в одинаковом порядке)*.
- **Чего нет**: Никаких ограничений на перестановку других инструкций. Не создаёт отношения _synchronizes-with_.
- **Применение**: Независимые счётчики *(например, статистика, метрики)*, где важна точность инкремента, но не важен порядок выполнения окружающего кода.

### `std::memory_order_release` / `std::memory_order_acquire`

- `memory_order_release` *(для операций записи)*: Ни одна операция записи или чтения в текущем потоке, находящаяся в коде _до_ этой инструкции, не может быть переставлена компилятором или процессором _после_ неё.
- `memory_order_acquire` *(для операций чтения)*: Ни одна операция записи или чтения в текущем потоке, находящаяся в коде _после_ этой инструкции, не может быть переставлена компилятором или процессором _до_ неё.

#### Пример паттерна Message Passing:
```C++
#include <atomic>
#include <thread>
#include <cassert>
#include <string>

std::string data; // Неатомарные данные
std::atomic<bool> ready{false};

void producer() {
    data = "Important Payload"; // 1 (sequenced-before 2)
    ready.store(true, std::memory_order_release); // 2
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)); // 3
    // 2 (release) synchronizes-with 3 (acquire)
    // Следовательно: 1 happens-before 4
    assert(data == "Important Payload"); // 4 (Никогда не сработает)
}
```

### `std::memory_order_consume`

- Создаёт отношение `dependency-ordered happens-before`. Ограничивает перестановки только для тех инструкций, которые имеют прямую **зависимость по данным** от загруженного значения *(например, чтение по указателю, который только что считали)*.
- На текущий момент во всех популярных компиляторах (GCC, Clang) спецификация `consume` временно переводится в полноценный `acquire`, т.к. компиляторам тяжело отслеживать цепочки зависимостей на этапе оптимизации регистров.

### `std::memory_order_acq_rel`

Применяется исключительно для операций типа **Read-Modify-Write (RMW)**, таких как `fetch_add`, `exchange`, `compare_exchange`.
- Объединяет обе семантики: данная инструкция служит как `acquire` для операции чтения и как `release` для операции записи.

### `std::memory_order_seq_cst`

- Помимо обеспечения Acquire-Release упорядочивания, гарантирует существование **единого глобального порядка** для всех `seq_cst` операций во всей программе.
- Все потоки видят выполнение `seq_cst` инструкций в абсолютно одинаковой последовательности.

## Барьеры памяти

Разные процессорные архитектуры предоставляют разные аппаратные гарантии консистентности памяти:

- **Weak Memory Model (ARM, PowerPC)**: Процессор на уровне железа может переставлять любые типы операций: Чтение-Чтение (`LoadLoad`), Чтение-Запись (`LoadStore`), Запись-Запись (`StoreStore`), Запись-Чтение (`StoreLoad`).
- **Strong Memory Model (x86-64 / TSO - Total Store Order)**: Процессор x86-64 строго гарантирует порядок для `LoadLoad`, `LoadStore` и `StoreStore`. Однако он **допускает одну единственную аппаратную перестановку: StoreLoad**.

### Проблема Store Buffering в x86-64

В каждом ядре процессора x86-64 есть локальный аппаратный буфер записи (**Store Buffer**). Когда поток выполняет запись в память, данные сначала пишутся в Store Buffer *(чтобы ядро не ждало медленной синхронизации с кэшем L1/L3)*. Последующая инструкция чтения из другого адреса может выполниться процессором спекулятивно прямо из L1 кэша _до_ того, как данные из Store Buffer физически сбросятся в общую систему кэшей. Это наглядно иллюстрируется классическим **Алгоритмом Деккера/Петерсона**:
```C++
#include <atomic>
#include <thread>
#include <cassert>

std::atomic<int> x{0};
std::atomic<int> y{0};
int r1 = 0;
int r2 = 0;

void thread1() {
    x.store(1, std::memory_order_relaxed);  // Store X
    r1 = y.load(std::memory_order_relaxed); // Load Y
}

void thread2() {
    y.store(1, std::memory_order_relaxed);  // Store Y
    r2 = x.load(std::memory_order_relaxed); // Load X
}

int main() {
    std::thread t1(thread1);
    std::thread t2(thread2);
    t1.join(); t2.join();
    
    // На архитектуре x86-64 из-за Store Buffer ВОЗМОЖНА ситуация, 
    // когда r1 == 0 и r2 == 0 одновременно!
    // Каждое ядро сначала выполнило свой Load, пока Store "застрял" в Store Buffer.
}
```

### Барьеры памяти

Для предотвращения подобных перестановок модель памяти C++ предоставляет явные барьеры `std::atomic_thread_fence()`, которые транслируются в специализированные инструкции процессора.

#### Аппаратные барьеры x86:

1. **`mfence` (Memory Fence)**: Полный барьер памяти. Он полностью останавливает выполнение последующих инструкций чтения и записи до тех пор, пока локальный `Store Buffer` ядра процессора не будет полностью опустошен в кэш-линии. Именно инструкция `mfence` предотвращает аппаратную гонку `StoreLoad`.

2. **`sfence` (Store Fence)**: Упорядочивает только инструкции записи (`StoreStore`). На x86-64 применяется редко (т.к. TSO и так держит порядок записей), в основном при работе с некэшируемой памятью.

3. **`lfence` (Load Fence)**: Упорядочивает только чтения (`LoadLoad`) и выполняет сериализацию конвейера, запрещая спекулятивное выполнение инструкций за барьер.

### Как `std::atomic` транслируется в ассемблер x86-64?

Благодаря сильной модели памяти x86-64 (TSO), компиляторам не нужно генерировать дорогие барьеры для Acquire-Release семантики:
- `load(memory_order_acquire)` превращается в обычную инструкцию **`mov`** (так как LoadLoad и LoadStore перестановки запрещены аппаратно).
- `store(memory_order_release)` превращается в обычную инструкцию **`mov`** (так как StoreStore и LoadStore перестановки запрещены аппаратно).
- Операции **`memory_order_seq_cst`** требуют предотвращения StoreLoad. Поэтому:
    - Запись `store(seq_cst)` компилятор транслирует либо в комбинацию **`mov + mfence`**, либо использует атомарную инструкцию обмена **`xchg`**, которая на x86 автоматически несёт в себе аппаратный префикс `lock` и выполняет функции полного барьера.
    - Любая атомарная RMW-операция компилятором транслируется с префиксом **`lock`** (например, `lock cmpxchg` / `lock xadd`), что аппаратно блокирует шину кэша и гарантирует полную последовательную консистентность.
