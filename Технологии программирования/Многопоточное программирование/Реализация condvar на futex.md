## 1. Наивная реализация

Если попытаться написать [[Condvar|condvar]] тривиально, используя системный вызов [[Futex|futex]], мы получим классическое состояние гонки. Представим наивный алгоритм `Wait`:

1. Поток видит, что условие не выполнено.
2. Он вызывает `lock.unlock()`, освобождая mutex.
3. Он вызывает `futex_wait(&cv_word, 0)`.

#### Сценарий гонки:
1. Поток А выполняет шаг 2 (отпускает мьютекс), но ещё не успевает войти в ядро ОС для вызова `futex_wait`.
2. В этот микросекундный момент Поток Б захватывает мьютекс, меняет предикат и вызывает `NotifyOne`, который транслируется в `futex_wake(&cv_word, 1)`.
3. Так как Поток А ещё не спит в ядре, очередь ожидания futex пуста. Сигнал пробуждения уходит «в никуда».
4. Поток А возвращает себе управление, заходит в ядро и вызывает `futex_wait`. Он засыпает **навсегда**, хотя данные для него уже готовы.

## 2. Механизм счётчика поколений

Чтобы исключить утерянное пробуждение, CV обязана отслеживать «номера поколений» *(sequence numbers)* нотификаций. Для этого заводится переменная `std::atomic<uint32_t> generation_{0}`.

```c++
class Condvar {
private:
    std::atomic<uint32_t> generation_{0}; // Глобальный счётчик поколений

public:
    template <typename Mutex>
    void Wait(Mutex& lock) {  // метод вызывается под lock.Lock
        uint32_t current_gen = generation_.load(std::memory_order_relaxed);

        lock.Unlock();
        // Если перед syscall сменилось поколение, метод вернёт ошибку EWOULDBLOCK
        futex_wait(&generation_, current_gen);
        lock.Lock();
    }

    void NotifyOne() {        // метод вызывается под lock.Lock
        generation_.fetch_add(1, std::memory_order_seq_cst);
        futex_wake(&generation_, 1);
    }

    void NotifyAll() {
        generation_.fetch_add(1, std::memory_order_seq_cst);
        futex_wake(&generation_, INT_MAX);
    }
};
```

## 3. Оптимизация производительности

Если поток вызывает `cv.notify_one()` внутри критической секции, возникает **проблема Mesa-style scheduling**:
- Проснувшийся в ядре ждущий поток мгновенно пытается захватить mutex, но mutex всё ещё держится нотифицирующим потоком!
- Ждущий поток бьётся о закрытый замок mutex и тут же снова блокируется внутри ОС (уходит спать в очередь мьютекса). Происходят два лишних и дорогостоящих переключения контекста.

**Решение (`FUTEX_CMP_REQUEUE`):**
Современные condvar в Linux используют системный вызов `futex` с флагом `FUTEX_CMP_REQUEUE`. Вместо того чтобы физически будить потоки из очереди condvar, ядро ОС **напрямую перевешивает** спящие потоки из очереди ожидания условной переменной в очередь ожидания mutex внутри самого ядра. Потоки просыпаются только тогда, когда mutex реально освобождается.
