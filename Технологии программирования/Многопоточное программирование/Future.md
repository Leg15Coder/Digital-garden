В традиционном синхронном программировании вызов функции блокирует поток до тех пор, пока результат не будет вычислен. В конкурентном программировании для разделения во времени момента **запуска** задачи и момента **получения её результата** используется пара примитивов: `Future` и `Promise`.

==Опр.== **Future** - это объект-клиент, предоставляющий интерфейс для доступа к результату асинхронной операции, который, возможно, ещё не вычислен.
Основные методы концептуальной `Future`:
- `IsReady()` - возвращает `true/false`, готова ли задача *(вычисляется мгновенно без блокировки)*.
- `Wait()` - синхронно блокирует текущий поток до тех пор, пока результат не станет доступен.
- `GetResult()` *(или `.get()`)* - возвращает вычисленное значение *(если оно не готово, выполняет `Wait()` пока оно не будет готово)*.

==Опр.== **Promise** - это объект-сервер, представляющий собой принимающий конец канала связи. Поток, выполняющий асинхронную задачу, использует `Promise` для записи результата или ошибки.
Основные методы концептуального `Promise`:
- `SetValue(value)` - сохраняет вычисленный результат и сигнализирует ассоциированной `Future` о готовности.
- `SetException(exception_ptr)` - сохраняет информацию об ошибке/исключении, если в процессе выполнения задачи что-то пошло не так.

> [!tip] Применение Future для асинхронного выполнения задач
> Когда поток вызывает функцию, она не блокирует его, а мгновенно создаёт `Promise`, извлекает из него связанную `Future` через метод `getFuture()` и возвращает её вызывающему коду. Сама тяжёлая задача уходит на выполнение в фоновый поток. Вызывающий поток может параллельно выполнять другую полезную работу, а затем, когда результат действительно понадобится, вызвать метод получения значения.

## Реализация классов `std::future` и `std::promise`

Классы `std::promise` и `std::future`, появившиеся в стандарте C++11, внутри себя не содержат данных напрямую. Они являются легковесными обёртками-указателями на скрытый разделяемый объект, распределённый в динамической памяти - **Shared State**.

Shared State инкапсулирует в себе:
```c++
template<typename T>
struct SharedState {
	std::variant<std::optional<T>> value;
	std::variant<std::exception_ptr> exception;
	std::atomic<bool> is_ready;
	std::mutex mtx;
	std::condition_variable cv;
}
```

#### Алгоритм взаимодействия:

- Когда рабочий поток вызывает `promise.set_value()`, он захватывает mutex внутри Shared State, записывает данные, выставляет флаг `is_ready = true` и делает `cv.notify_all()`.
- Когда клиент вызывает `future.get()`, он захватывает этот же mutex и проверяет флаг. Если данные не готовы, он засыпает на `cv.wait()`.

> [!fail] Недостатки `std::future`
> 1. **Полное отсутствие композиции:**
> 	В стандартных `std::future` (вплоть до C++23) **нет** методов `.then()`, `when_all` или `when_any`! Из-за этого нельзя построить неблокирующий асинхронный pipeline.
>     
> 2. **Тяжеловесность:**
> 	Каждая связка promise/future вынуждена аллоцировать Shared State в куче и таскать за собой тяжёлые системные mutexes и [[Condvar|condvars]] ОС. Если в системе крутятся миллионы мелких асинхронных тасок, аллокации и блокировки на системных примитивах создают огромную нагрузку.
>     
> 3. **Одноразовость:**
>     Можно вызвать метод `.get()` на объекте `std::future` только **один раз**. Метод `.get()` перемещает *(move-семантика)* значение из Shared State, после чего состояние разрушается, а повторный вызов приводит к исключению.
>     
> 4. **Блокирующий деструктор `std::async`:**
>     Если `std::future` получена из вызова `std::async(std::launch::async, ...\)`, то её деструктор обладает специфическим поведением: он блокирует текущий поток до завершения работы асинхронной задачи, если результат ещё не забран. Это часто ломает ожидаемую асинхронность и приводит к скрытым дедлокам или просадкам производительности.

## Проблема Callback Hell

==Опр.== **Callback** - функция, которая передаётся в другую функцию и вызывается автоматически после завершения какого-то определённого действия.

==Опр.== **Callback Hell** - проблема глубокой вложенности кода при выполнении последовательности зависимых асинхронных операций.

```c++
// Пример Callback Hell
WhenRpcCallFinished(input, [](Output output) {
    ReadFileAsynchronously(output.filePath, [](Buffer buffer) {
        WriteToDatabaseAsynchronously(buffer, [](Status status) {
            std::cout << "Успешно завершено!" << std::endl;
        });
    });
});
```

### Ключевые недостатки Callback Hell

1. **Инверсия управления и структуры кода:** Весь код оказывается «вывернутым наизнанку». Вместо линейной последовательности действий код растёт вширь, напоминая пирамиду. Это критически ухудшает читаемость.
2. **Сложность обработки ошибок:** Классические исключения (`try-catch`) перестают работать, поскольку callbacks выполняются в других потоках выполнения. Разработчик вынужден вручную пробрасывать объекты ошибок через параметры каждого callback на каждом уровне вложенности.
3. **Невозможность нормальной декомпозиции:** Логику трудно разбить на отдельные переиспользуемые функции.

## Future Combining

==Опр.== **Continuation** - это возможность «подвесить» callback прямо на объект `Future` с помощью метода `.then()` или `.thenValue()`. Функция-callback автоматически выполнится, как только `Future` перейдет в состояние готовности.

==Опр.== **Chaining** - последовательная цепочка вызовов Continuation без вложенности. 

```c++
// Решение Callback Hell с помощью Chaining
Future<Output> f1 = SomeRpcCall(input);
Future<Status> f2 = std::move(f1)
    .thenValue([](Output output) {
        return ReadFileAsynchronously(output.filePath); // возвращает Future<Buffer>
    })
    .thenValue([](Buffer buffer) {
        return WriteToDatabaseAsynchronously(buffer);   // возвращает Future<Status>
    });
```

==Опр.== **Combining** - комбинирование или параллельная композиция нескольких задач.

#### Примеры Combining

1. **`when_all` (или `collect_all`):**
	 - Возвращает одну агрегированную `Future`, которая перейдет в состояние готовности **только тогда, когда ВСЕ** входящие future завершатся (неважно - успешно или с ошибкой).

2. **`when_any` (или `collect_any`):**
    - Она становится готовой, как только завершится **хотя бы одна (самая первая)** из входящих задач.

## Executors

==Опр.== **Executor** - это сущность, которая берёт на себя управление тем, ***где и как*** этот код будет физически выполнен *(на каком ядре, в каком потоке или контексте)*.

Без executor использование цепочек `.then()` становится опасным: callback, скорее всего, выполнится в контексте того потока, который вызвал `promise.set_value()`. Это может нарушить потокобезопасность или заблокировать критически важный поток.

Базовый интерфейс обычно представляет собой структуру с чистой виртуальной функцией `Execute`:
```c++
struct IExecutor {
    using Task = std::function<void()>;
    virtual void Execute(Task task) = 0; // Принимает задачу и планирует её выполнение
    virtual ~IExecutor() = default;
};
```

### Основные типы executors

1. **`ThreadedExecutor`:**
    При каждом вызове `Execute` лениво создаёт новый физический поток ОС (`std::thread`\), выполняет в нём задачу и завершает поток. Обладает колоссальной проблемой на частое создание потоков, поэтому используется редко.
    
2. **`ThreadPoolExecutor`:**
	При старте создаёт фиксированное или динамическое количество постоянно живущих рабочих потоков. Внутри содержит потокобезопасную (блокирующую) очередь задач. Потоки-работники забирают задачи из очереди и выполняют их, исключая накладные расходы на постоянное создание новых потоков.
    
3. **`SerializedExecutor`:**
    Специальный декоратор над другим executor, который гарантирует, что отправленные в него задачи будут выполняться **строго последовательно**, одна за другой.
    - _Зачем нужен:_ Позволяет организовать работу с разделяемым состоянием какого-то объекта без явного использования мьютексов *(нет параллельного доступа $\to$ нет Data Race)*.

Связующим звеном между продвинутыми future и executors является метод `.via(IExecutor*)`. Он осуществляет переключение контекста выполнения.
```c++
folly::ThreadPoolExecutor pool(4);
Promise<int> p;

// Привязываем будущую цепочку выполнения к пулу потоков pool
Future<int> f = p.getSemiFuture().via(&pool); 

auto f2 = std::move(f).thenValue([](int result) {
    // Этот код гарантированно выполнится внутри одного из потоков pool, 
    // независимо от того, какой поток вызовет p.setValue()!
    return result * 2;
});
```
