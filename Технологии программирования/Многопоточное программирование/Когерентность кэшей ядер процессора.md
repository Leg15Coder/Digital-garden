Современные процессоры работают значительно быстрее, чем оперативная память. Это явление называют **«стеной памяти»**. Чтобы процессор не простаивал в ожидании данных из RAM, между ядрами и памятью встроена сверхбыстрая кэш-память нескольких уровней (L1, L2 - обычно приватные для каждого ядра, L3 - общая). Данные из памяти в кэш копируются не по одному байту, а фиксированными блоками - **кэш-линиями**, размер которых на архитектурах x86/ARM почти всегда равен 64 байтам.

==Опр.== **Когерентность кэшей** - это аппаратное свойство системы, гарантирующее, что все ядра процессора видят единую, актуальную и непротиворечивую картину оперативной памяти. Аппаратное обеспечение должно координировать действия ядер так, чтобы изменения, сделанные одним ядром, мгновенно становились известны остальным.

Проблема **когерентности кэшей** возникает в многоядерных системах. Если Ядро №1 и Ядро №2 загрузили из RAM одну и ту же кэш-линию, содержащую переменную `X = 5`.
- Ядро №1 меняет `X` на `10` в своём локальном кэше L1.
- Если Ядро №2 прямо сейчас попытается прочитать `X` из своего кэша L1, оно увидит старое значение `5`.

==Опр.== **Протокол MESI** - специальный протокол, используемый для обеспечения когерентности архитектуры процессоров.
#### Четыре состояния MESI:

1. **M - Modified (Изменена)**
    - **Суть:** Данная кэш-линия присутствует _только в кэше текущего ядра_, и данные в ней были изменены (она «грязная»). Копия в оперативной памяти (RAM) устарела.
    - **Действие:** Ядро может свободно читать и писать в эту линию. Если другое ядро захочет прочитать эти данные, текущее ядро обязано сначала переслать их ему (или сбросить в RAM).
    
2. **E - Exclusive (Эксклюзивная)**
    - **Суть:** Кэш-линия присутствует _только в кэше текущего ядра_, но данные в ней «чистые» - они _полностью совпадают_ с данными в основной памяти (RAM).
    - **Действие:** Ядро может читать её. Если ядро решает записать данные в эту линию, она мгновенно и без лишних сигналов по межъядерной шине переходит в состояние **Modified (M)**.
    
3. **S - Shared (Разделяемая)**
    - **Суть:** Кэш-линия присутствует в кэшах _сразу нескольких ядер_. Данные во всех кэшах чистые и совпадают с RAM.
    - **Действие:** Любое из этих ядер может свободно _читать_ линию. Но если одно из ядер захочет выполнить _запись_, оно обязано отправить по межъядерной шине сигнал инвалидации (**Invalidate**), требуя от всех остальных ядер выбросить эту линию из своих кэшей. Только после этого пишущее ядро переводит линию в состояние **M**.
    
4. **I - Invalid (Недействительная)**
    - **Суть:** Кэш-линия пуста или содержит устаревшие данные.
    - **Действие:** Попытка чтения или записи приведет к промаху кэша (**Cache Miss**), и процессору придётся запрашивать актуальную кэш-линию у других ядер или из RAM.
    

#### Пример жизненного цикла кэшей:

1. **Ядро 1** читает переменную `A` из RAM. Линия загружается в состоянии **E** (я один владею ей).
2. **Ядро 2** тоже читает переменную `A`. Теперь у обоих ядер эта линия переходит в состояние **S** (мы делимся ей).
3. **Ядро 1** решает записать новое значение в `A`. Оно посылает сигнал по шине: _«Всем удалить эту линию!»_. У **Ядра 2** состояние падает в **I**. У **Ядра 1** состояние становится **M** (я изменил её).

### Эффект Cache Ping-Pong

==Опр.== **Cache Ping-Pong** - это резкое падение производительности многопоточного приложения, вызванное тем, что одна и та же кэш-линия непрерывно «прыгает» между кэшами разных ядер процессора.

Это может произойти когда 2 потока на разных ядрах интенсивно изменяют данные, находящиеся в пределах одной 64-байтной кэш-линии.
1. **Ядро А** пишет в линию. Линия переходит в состояние **Modified (M)** в кэше А. В кэше **Ядра Б** эта линия принудительно переводится в **Invalid (I)**.
2. Спустя наносекунду **Ядро Б** пытается сделать свою запись. Оно видит состояние **I**. Процессор приостанавливает выполнение, запрашивает линию у Ядра А. Ядро А вынуждено отдать линию. Теперь у Ядра Б состояние становится **M**, а у Ядра А - **I**.
3. Ещё через наносекунду **Ядро А** снова пишет туда же... история повторяется.

Каждый такой «прыжок» линии между ядрами через межсистемную шину занимает от 20 до 100 наносекунд. Процессоры тратят колоссальное количество времени на аппаратное согласование кэшей и пересылку данных вместо выполнения полезных вычислений. Кэш-пинг-понг может возникать как при **True Sharing** *(когда потоки реально разделяют одну переменную, например, атомарный счетчик или мьютекс)*, так и при **False Sharing**.

### Проблема False Sharing

==Опр.== **False Sharing** - это ситуация, когда с точки зрения логики программы потоки абсолютно независимы и работают с разными, не пересекающимися переменными, но из-за неудачного расположения в памяти эти переменные оказываются внутри **одной и той же кэш-линии (64 байта)**.

Пример. У нас есть массив из двух структур для двух потоков:
```C++
struct ThreadData {
    uint64_t counter = 0; // Занимает 8 байт
};
ThreadData data[2]; // data[0] и data[1] расположены в памяти вплотную
```

Логически Поток 0 инкрементирует только `data[0].counter`, а Поток 1 - только `data[1].counter`. Между ними нет состояния гонки, код абсолютно потокобезопасен, нам даже не нужны мьютексы! **Однако на уровне железа:**

Размер одной структуры = 8 байт. Обе структуры суммарно занимают 16 байт и гарантированно попадают в **одну 64-байтную кэш-линию**. Когда Поток 0 на Ядре 0 инкрементирует свой счётчик, протокол MESI инвалидирует _всю кэш-линию целиком_ для Ядра 1. Поток 1 сталкивается с **I**, ждёт пересылки кэш-линии, инкрементирует свой счётчик, тем самым инвалидируя линию для Ядра 0. В итоге независимые потоки начинают неявно блокировать друг друга на уровне железа, порождая описанный выше **Cache Ping-Pong**. Скорость работы программы может упасть в разы по сравнению даже с однопоточной версией.

Все решения сводятся к одному правилу: **разнести независимые переменные по разным кэш-линиям**:
#### Способ 1. Использование выравнивания (`alignas`)

Начиная с C++17, в стандарте появились константы выравнивания, учитывающие архитектуру процессора (находятся в заголовке `<new>`). Для борьбы с false sharing используется `std::hardware_destructive_interference_size` (обычно равна 64 байтам). Мы можем принудительно выровнять структуру по размеру кэш-линии:
```C++
#include <new>
#include <cstdint>

struct alignas(std::hardware_destructive_interference_size) ThreadData {
    uint64_t counter = 0;
};

// Теперь data[0] и data[1] будут гарантированно лежать в разных кэш-линиях.
// Между ними в памяти компилятор сделает автоматический отступ.
ThreadData data[2]; 
```

#### Способ 2. Ручной Padding

Если под рукой нет C++17, можно вручную добавить пустые байты («отступы»), чтобы размер структуры стал равен или кратен 64 байтам:
```C++
struct ThreadData {
    uint64_t counter = 0; // 8 байт
    // Добиваем размер структуры до 64 байт
    char padding[64 - sizeof(uint64_t)]; 
};
```

#### Способ 3. Использование локальных переменных

Самый элегантный и часто наиболее производительный способ - вообще не трогать разделяемую память во время тяжелых вычислений. Вместо того чтобы внутри цикла постоянно писать в общий массив, поток должен накапливать результат в локальной переменной (которая будет лежать в регистре процессора или на приватном стеке потока), а финальный результат записать в глобальную структуру **всего один раз** в самом конце работы:

```C++
// ПЛОХО (Вызывает False Sharing):
void worker(int thread_id) {
    for (int i = 0; i < 10'000'000; ++i) {
        data[thread_id].counter += calculate(i); 
    }
}

// ХОРОШО (Кэш не страдает):
void worker_optimized(int thread_id) {
    uint64_t local_counter = 0; // Живет на стеке/в регистре ядра
    for (int i = 0; i < 10'000'000; ++i) {
        local_counter += calculate(i);
    }
    data[thread_id].counter = local_counter; // Единственная запись в конце
}
```
