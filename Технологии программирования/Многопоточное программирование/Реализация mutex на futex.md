[[Spinlock]] сжигают 100% CPU, если критическая секция долгая. Для долгих задач нужен mutex, который взаимодействует с операционной системой через системный вызов [[Futex]], усыпляя потоки в ядре ОС, когда критическая секция занята.

## Реализации
### 1. Двухпозиционный Futex-mutex

Если строить мьютекс на простом булевском флаге (0 - свободен, 1 - занят), то при каждом вызове `unlock()` мы будем обязаны вызывать системный вызов `futex_wake`, чтобы проверить, нет ли спящих. 

```c++
class NaiveFutexMutex {
private:
    std::atomic<int> state_{0}; // 0 - unlocked, 1 - locked
public:
    void Lock() {
        // Крутимся, пока не сможем атомарно перевести из 0 в 1
        while (state_.exchange(1) == 1) {
            // Уходим спать в ядро ОС. 
            futex_wait(&state_, 1); 
        }
    }
    void Unlock() {
        state_.store(0);
        // Сискол wake вызывается ВСЕГДА!
        futex_wake(&state_, 1); 
    }
};
```

- **Как futex защищает от Lost Wake-up:**
    _Сценарий гонки:_ Поток Б проверил `while`, замок занят. Поток Б собирается вызвать `futex_wait`. Прямо перед системным вызовом контекст переключается. Поток А делает `Unlock()`: сбрасывает `state_` в `0` и делает `futex_wake`. Очередь пуста, сигнал улетает в никуда. Контекст возвращается к Потоку Б, и он вызывает `futex_wait(&state_, 1)`. Системный вызов `futex_wait` внутри ядра атомарно перепроверяет: `if (*addr == val)`. Ядро видит, что в памяти по адресу `&state_` уже лежит 0, а мы передали ожидаемое значение 1. Проверка проваливается, ядро мгновенно возвращает управление Потоку Б с ошибкой `EWOULDBLOCK`, не укладывая его спать. Поток Б делает следующий круг цикла и забирает замок.
    
- **Критический недостаток этой реализации:**
    В методе `Unlock()` системный вызов `futex_wake` происходит **при каждом освобождении мьютекса**, даже если за замком никто не стоит. Теряется весь смысл концепции "Fast Userspace", где в отсутствие конкуренции всё должно работать без syscall.

### 2. Mutex на 3 состояниях

Чтобы полностью исключить системные вызовы на `Unlock()`, если конкуренции нет, состояние мьютекса расширяют до трёх значений:
- **`0` (Unlocked):** Мьютекс полностью свободен.
- **`1` (Locked, no waiters):** Мьютекс занят одним потоком, но за ним в очереди больше никто не стоит.
- **`2` (Locked, with waiters):** Мьютекс занят, и есть другие потоки, которые заблокировались и ушли спать в ядро.

#### Как это оптимизирует работу?

- **При захвате (Lock):** Поток пытается атомарно заменить `0` на `1`. Если получилось - мы захватили блокировку без syscall. Если там уже была `1` или `2`, поток переводит состояние в `2` (указывая, что теперь точно есть ждущие) и вызывает `FUTEX_WAIT`.

- **При освобождении (Unlock):** Поток атомарно уменьшает значение или делает CAS.
    - Если состояние было `1` (никто не ждал), поток просто сбрасывает его в `0`. **Системный вызов `FUTEX_WAKE` не вызывается!** Мы сэкономили кучу времени.
    - Если состояние было `2` (кто-то спит в ядре), поток сбрасывает его в `0` и **обязательно вызывает `FUTEX_WAKE`**, чтобы разбудить спящего товарища.

#### Код реализации
```c++
class FutexMutex3States {
private:
    std::atomic<int> state_{0};

public:
    void Lock() {
        int expected = 0;
        // Если mutex полностью свободен, атомарно переводим его в состояние 1
        if (state_.compare_exchange_strong(expected, 1, std::memory_order_acquire)) {
            return;
        }

        // Встаём в очередь, т.к. mutex уже занят
        if (expected != 2) {
            expected = state_.exchange(2, std::memory_order_acquire);
        }

        // Входим в цикл ожидания
        while (expected != 0) {
            // Уходим спать в ядро ОС
            futex_wait(&state_, 2);
            
            // После пробуждения пытаемся снова захватить мьютекс
            expected = state_.exchange(2, std::memory_order_acquire);
        }
    }

    void Unlock() {
        // Атомарно освобождаем замок, записывая 0, и смотрим на предыдущее состояние.
        // Если оно было 1, значит никто не пришёл и не встал в очередь.
        // Мы просто обнулили переменную и сэкономили кучу времени на системном вызове!
        // Если же оно было 2, значит, в ядре гарантированно есть спящие потоки.
        if (state_.exchange(0, std::memory_order_release) == 2) {
            // Вызываем пробуждение для 1 потока
            futex_wake(&state_, 1);
        }
    }
};
```

## Низкоуровневые сценарии конкуренции
### Сценарий 1: Быстрый путь
1. **Захват (`Lock`):** Поток А приходит к свободному мьютексу (переменная равна `0`). Он выполняет атомарную RMW-операцию, пытаясь установить значение `1`. Операция возвращает старое значение `0`. Поток А понимает, что замок успешно взят, и мгновенно заходит в критическую секцию. 
2. **Освобождение (`Unlock`):** Поток А закончил работу. Он атомарно проверяет состояние. Поток А видит, что состояние мьютекса по-прежнему равно `1` (никто не пришел, пока он работал). Он просто сбрасывает состояние в `0`. Мы получили скорость spinlock в отсутствие конкуренции!

### Сценарий 2: Медленный путь (Появление конкурента)
1. Поток А удерживает мьютекс (состояние равно `1`).
2. Приходит Поток Б, выполняет атомарный `exchange(1)`. Назад возвращается `1`. Поток Б понимает: _«Замок занят Потоком А»_.
3. Поток Б осознает, что ему нужно идти спать в ядро. Но перед этим он обязан гарантировать, что поток А узнает о его присутствии. Поток Б с помощью атомарной операции принудительно переводит состояние мьютекса в `2`.
4. Поток Б вызывает `futex_wait(addr, 2)`. Ядро Linux проверяет: _«Правда ли там сейчас лежит двойка?»_. Да, поток Б только что записал её. Ядро усыпляет Поток Б и забирает у него управление процессором.
5. Поток А выходит из критической секции. Делает `state_.exchange(0)`. Ему возвращается старое значение `2`. Поток А понимает: _«За мной заняли очередь в ядре»_. Он обязан вызвать `futex_wake(&state_, 1)` для пробуждения Потока Б.

### Сценарий 3: Феномен накатывания
1. Поток А вызвал `unlock`, сбросил состояние в `0` и сделал `futex_wake`. Поток Б (который спал) начинает процедуру пробуждения в недрах ОС.
2. Пробуждение из ядра занимает время *(планировщику нужно выделить квант времени, переключить контекст)*. Поток Б ещё физически не выполнил ни одной строчки кода в Userspace.
3. В этот самый микросекундный момент на процессорном ядре прорывается абсолютно новый, активный **Поток В**. Он вызывает метод `lock()`, делает атомарный `exchange` и видит, что состояние мьютекса равно `0`.
4. Поток В мгновенно захватывает мьютекс, меняет состояние на `1` (или `2`) и уходит выполнять критическую секцию.
5. **Что происходит со спящим Потоком Б, когда он окончательно «открыл глаза»:** Поток Б выходит из системного вызова `futex_wait`. Он заходит на следующую итерацию своего внутреннего цикла ожидания `while` и пытается атомарно забрать замок. Но он видит, что значение уже НЕ 0.
6. Поток Б понимает, что замок опять занят, атомарно выставляет состояние мьютекса обратно в `2` и послушно уходит спать назад в `futex_wait`.

_Такое поведение называется **Barging (накатывание / нечестный мьютекс)**. Мьютекс получается нечестным (Поток Б ждал долго, а Поток В проскочил без очереди), но это колоссально увеличивает общую пропускную способность системы. Процессор не простаивает ни одного такта, пока Поток Б медленно просыпается из ядра._
