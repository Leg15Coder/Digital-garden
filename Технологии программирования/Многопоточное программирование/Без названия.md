**Реализация многопоточного сервера через epoll (Паттерн Reactor):**

Один выделенный поток (Reactor) крутит цикл с `epoll_wait()`. Как только `epoll_wait` возвращает список "готовых" дескрипторов, Reactor не обрабатывает их сам, а формирует задачи (например, `ReadTask(fd)`) и кладет их в очередь Thread Pool. Рабочие потоки из пула достают задачу, делают один успешный (неблокирующий) `read`, так как epoll уже гарантировал наличие данных, и отправляют ответ.

### 12. Механизм io_uring

**Проблема `epoll`:** Это **синхронный** механизм. `epoll` только говорит: _"Данные есть, иди читай"_. Потоку всё равно приходится делать сисколл `read`, переключать контекст в режим ядра, чтобы ядро скопировало данные из буфера сетевой карты в буфер приложения.

`io_uring` — это **истинно асинхронный** ввод-вывод, появившийся в ядре Linux 5.1.

Здесь мы говорим ядру: _"Вот буфер в user-space. Самостоятельно считай данные из этого сокета сюда и сообщи мне, когда закончишь"_.

**Архитектура (Как это работает):**

Основана на двух кольцевых буферах (Ring Buffers), память которых расшарена (shared memory) между приложением (user-space) и ядром (kernel-space).

1. **SQ (Submission Queue):** Приложение помещает сюда заявки SQE (Submission Queue Entry). Например: "Прочти 1024 байта из сокета 5 в буфер X".
    
2. **CQ (Completion Queue):** Ядро выполняет работу в фоне и кладет результат CQE (Completion Queue Entry) сюда: "Прочитано 512 байт / Ошибка соединения".
    

**Преимущества io_uring перед epoll:**

- **Zero Syscalls:** Благодаря shared memory можно отправлять сотни запросов и забирать сотни результатов _вообще без системных вызовов_ (переключений контекста), что дает колоссальную пропускную способность.
    
- **Унифицированность:** `epoll` ужасно работает с обычными файлами на диске (в Linux диск почти всегда считается "готовым", поэтому `epoll` превращается в обычный блокирующий `read`). `io_uring` идеально и единообразно асинхронно работает как с сетью, так и с диском.
    

**Недостатки:**

- Требует относительно новых версий ядра ОС.
    
- Сложнее в настройке и использовании (много ручной работы с барьерами памяти, если не использовать `liburing`).
    
- Имеет нюансы с безопасностью (в контейнерах Kubernetes/Docker доступ к io_uring часто по умолчанию заблокирован из-за прямого доступа к памяти ядра).
    

### 13. Примитивы Future и Promise. Callback Hell и композиция

Отправляя задачу в Thread Pool, возникает проблема: как главному потоку получить результат вычислений, если задача выполняется где-то в неизвестном рабочем потоке?

**Идея:** Разделить канал связи на два объекта:

- **Promise (Обещание):** "Продюсер". Ручка, которая передается в рабочий поток. Когда задача выполнена, поток делает `promise.set_value(result)`.
    
- **Future (Будущее):** "Консьюмер". Read-only ручка, которая остается у того, кто поставил задачу. Вызов `future.get()` вернет результат.
    

**Проблема блокировки и Callback Hell:**

Если главный поток вызовет `future.get()`, он заблокируется в ожидании! Асинхронность пропадает.

Исторически для обхода этого использовали **Коллбеки (Callbacks)**: передавали в пул функцию, которую нужно вызвать по завершении:

C++

```
thread_pool.submit(task_A, [](Result_A res_A) {
    thread_pool.submit(task_B(res_A), [](Result_B res_B) {
         // И так далее...
    });
});
```

Это порождает **Callback Hell** (ад коллбеков) — код "уезжает" вправо, логика разрывается на куски, а корректно отловить и передать исключение наверх становится нетривиальной задачей.

**Решение (Future continuations, chaining и combining):**

Современная парадигма требует, чтобы Future были "компонуемыми".

- **Continuations (`.then()`):** Позволяет привязать продолжение напрямую к объекту future. Метод возвращает новую Future.
    
- **Chaining (Сцепление):** Превращает ад коллбеков в плоский, читаемый сверху вниз пайплайн:
    
    C++
    
    ```
    async_read_db()
        .then([](Data d) { return process(d); })
        .then([](Result r) { return async_send_network(r); })
        .then([](Status s) { log("Success"); });
    ```
    
- **Combining (Слияние):** Механизмы синхронизации ветвлений.
    
    - `when_all(f1, f2)` — возвращает future, которая будет готова, когда выполнятся _все_ аргументы.
        
    - `when_any(f1, f2)` — когда выполнится _первая_ (идеально для реализации таймаутов).
        

### 14. Реализация классов std::future и std::promise. Их неудобства

**Как это работает "под капотом":**

Классы `std::promise` и `std::future` являются просто обертками-указателями на скрытый разделяемый объект — **Shared State** (Разделяемое состояние).

Этот Shared State создается динамически в куче (heap) при создании `promise` и содержит:

1. Хранилище для результата или `std::exception_ptr`.
    
2. Флаг готовности `is_ready`.
    
3. Примитивы синхронизации: `std::mutex` и `std::condition_variable`.
    

Когда рабочий поток делает `set_value()`, он лочит мьютекс, записывает данные, ставит флаг и делает `notify_all()` на кондваре. `future.get()` лочит мьютекс и, если флаг не выставлен, ложится спать через `wait()`.

**В чем их огромное неудобство в C++?**

1. **Нет композиции:** В стандартной библиотеке `<future>` (вплоть до последних стандартов) **НЕТ** методов `.then()`, `when_all` или `when_any`! С `std::future` вы можете только блокирующе ждать через `.get()` или поллить через `.wait_for()`. Из-за этого стандартные фьючи практически бесполезны для построения сложных асинхронных пайплайнов.
    
2. **Тяжеловесность:** Каждая связка promise/future вынуждена делать медленную аллокацию Shared State в куче и таскает за собой тяжелые системные мьютексы. Для миллионов мелких тасок это колоссальный оверхед.
    
3. **Одноразовость:** Метод `.get()` можно вызвать только один раз (значение перемещается, future инвалидируется).
    
4. **Отсутствие отмены:** Стандартный API не предоставляет механизма безопасной отмены (cancellation) уже запущенной асинхронной операции.
    

_Именно поэтому в Production (до появления корутин C++20) повсеместно использовались кастомные реализации из библиотек Boost, Folly или Seastar, которые поддерживают `.then()` и избавлены от жестких аллокаций._
