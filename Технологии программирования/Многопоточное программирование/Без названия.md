### 1.3. Концепция кооперативной многозадачности

Для решения дилеммы «удобный линейный код против высокой производительности» были разработаны механизмы **кооперативной многозадачности** (User-space concurrency).

В отличие от вытесняющей многозадачности (Preemptive multitasking), где операционная система может прервать поток выполнения в любой случайный момент времени, в кооперативной модели задача выполняется непрерывно до тех пор, пока сама добровольно не отдаст управление планировщику (через операцию `yield`) или не заблокируется на специализированном неблокирующем примитиве ожидания. Это позволяет писать код в привычном синхронном виде, сохраняя под капотом эффективность асинхронного Event Loop.


## 4. Тестирование конкурентности: Внедрение сбоев (Fault Injection)

### 4.1. Проблема недетерминизма многопоточного кода

Поиск ошибок параллельного программирования (Data Races, Deadlocks, Нарушения инвариантов критических секций) классическими методами малоэффективен. Планировщик операционной системы вытесняет потоки на основе аппаратных прерываний таймера, сетевой активности и нагрузки на CPU. Состояние гонки может проявляться один раз на миллион запусков в промышленной среде, но ни разу не воспроизвестись на машине разработчика. Использование `std::this_thread::sleep_for` замедляет тесты и не гарантирует покрытие всех сценариев.

### 4.2. Концепция детерминированного тестирования в user-space

Поскольку файберы функционируют полностью в пространстве пользователя, их переключение контролируется исключительно кодом планировщика. Это позволяет сделать выполнение многопоточного алгоритма **абсолютно детерминированным**.

Для этого применяется подход **Fault Injection (внедрение сбоев)**, используемый в специализированных академических библиотеках тестирования (например, в учебной библиотеке _Lines_ или _Twist_). В исходный код примитивов синхронизации (кастомные атомики, мьютексы, условные переменные) интегрируются вызовы инжектора:

```C++
// Код примитива синхронизации, адаптированный для тестирования
class TestMutex {
public:
    void Lock() {
        while (locked_.exchange(true)) {
            // В режиме тестирования принудительно вызываем инжектор сбоев
            InjectFault(); 
            // Реальная логика ожидания файбера...
        }
    }

    void Unlock() {
        locked_.store(false);
        InjectFault(); // Проверяем, что будет, если контекст переключится сразу после анлока
    }

private:
    void InjectFault() {
        // Обращаемся к глобальному инжектору тестирования
        if (TestEngine::FaultInjectionEnabled()) {
            TestEngine::GetNextDecision(); // Принимает детерминированное решение
        }
    }
    std::atomic<bool> locked_{false};
};
```

### 4.3. Алгоритмы инжектора и воспроизводимость

Инжектор сбоев (`TestEngine`) может работать в двух режимах:

1. **Рандомизированный детерминизм:** Инжектор принимает решение о вызове `Scheduler::Yield()` на основе псевдослучайного генератора (например, Вихря Мерсенна `std::mt19937`\). Если зафиксировать стартовое число генератора (**Seed**), то вся последовательность переключений файберов между критическими секциями повторится один в один при каждом перезапуске теста. Любой обнаруженный баг становится стопроцентно воспроизводимым.
    
2. **Полный перебор графа исполнений (Exhaustive Search):** Инжектор строит дерево всех возможных вариантов переплетения потоков выполнения (_Interleavings_). На каждом шаге, где файбер делает операцию с памятью, планировщик поочередно запускает то один файбер, то другой. Это позволяет математически гарантировать отсутствие состояний гонки на данном наборе входных данных.

## 6. Внутреннее устройство бесстековых корутин в C++20
### 6.2. Анатомия триады управления

Управление бесстековой корутиной распределено между тремя компонентами:

```
+-------------------------------------------------------------+
|                      COROUTINE FRAME                        |
|                                                             |
|  +------------------------+   +--------------------------+  |
|  |     Promise Object     |   | Local Variables / State  |  |
|  | (Интерфейс разработчика)|   |  (Данные автомата)       |  |
|  +------------------------+   +--------------------------+  |
+-------------------------------------------------------------+
                           ^
                           | Ссылается (void*)
                +-----------------------+
                |   Coroutine Handle    |
                | (std::coroutine_handle|
                +-----------------------+
```

1. **Coroutine Frame:** Внутренний блок памяти в куче, содержащий Promise, аргументы, локальные переменные и стейт-индекс.
2. **Promise Object:** Создается внутри фрейма. Это интерфейс, через который автор библиотеки настраивает поведение корутины. Обязан содержать методы `initial_suspend`, `final_suspend`, `get_return_object`, `unhandled_exception`, `return_value` или `return_void`.
3. **Coroutine Handle (`std::coroutine_handle<>`\):** Не владеющий указатель на фрейм, доступный пользователю.

Реализация кастомного класса `Task` с объектом Promise:
```C++
#include <coroutine>
#include <exception>
#include <iostream>

class Task {
public:
    struct promise_type;
    using handle_type = std::coroutine_handle<promise_type>;

    struct promise_type {
        // Вызывается компилятором первым делом для конструирования внешнего объекта Task
        Task get_return_object() {
            return Task(handle_type::from_promise(*this));
        }

        // Задаем поведение при старте: приостановить выполнение (Lazy evaluation)
        std::suspend_always initial_suspend() { return {}; }

        // Задаем поведение при завершении: приостановить, чтобы сохранить promise живым
        std::suspend_always final_suspend() noexcept { return {}; }

        // Сохраняем возвращаемое значение корутины
        void return_value(int value) {
            value_ = value;
        }

        // Перехватываем исключения, выброшенные в корутине
        void unhandled_exception() {
            exception_ = std::current_exception();
        }

        int value_ = 0;
        std::exception_ptr exception_;
    };

    Task(handle_type handle) : handle_(handle) {}
    
    ~Task() {
        if (handle_) handle_.destroy(); // Ручное освобождение фрейма в куче
    }

    void Resume() {
        if (handle_ && !handle_.done()) {
            handle_.resume(); // Передаем процессор корутине
        }
    }

    int GetResult() {
        if (handle_.promise().exception_) {
            std::rethrow_exception(handle_.promise().exception_);
        }
        return handle_.promise().value_;
    }

private:
    handle_type handle_;
};
```

## 7. Оператор `co_await` и протокол Авейтеров (Awaitables)

### 7.1. Низкоуровневый разбор протокола `co_await`

Когда в теле корутины вызывается выражение `co_await expr;`, компилятор C++20 разворачивает его в цепочку вызовов трех методов специального объекта - **Awaiter**. Псевдокод развёртывания выглядит следующим образом:
```C++
auto&& awaiter = get_awaiter(expr); // Получаем объект-awaiter

if (!awaiter.await_ready()) {
    // Корутина фиксирует текущую точку и переходит в состояние Suspended
    // Вызов suspend с передачей дескриптора текущей корутины
    awaiter.await_suspend(current_coroutine_handle); 
    // Управление возвращается тому потоку, который вызвал handle.resume()
}
// Вызывается при пробуждении. Результат становится результатом всего выражения co_await
auto result = awaiter.await_resume(); 
```

- **`bool await_ready()`:** Позволяет избежать приостановки, если асинхронная операция уже завершилась к моменту вызова. Например, если мы ждем данные из сокета, а они уже находятся в буфере ядра. Если метод возвращает `true`, корутина продолжает выполнение без засыпания.
    
- **`await_suspend(std::coroutine_handle<> handle\)`:** Метод вызывается, когда корутина уже гарантированно приостановлена. Внутри этого метода разработчик регистрирует `handle` в структурах данных Event Loop (например, связывает с системным дескриптором `epoll`).
    
- **`await_resume()`:** Возвращает результат выполнения асинхронной операции. Если асинхронная операция завершилась ошибкой, именно внутри `await_resume()` генерируется исключение C++, которое легитимно ловится внешним блоком `try-catch` в теле корутины.
    

### 7.2. Симметричная передача управления (Symmetric Transfer)

Если метод `await_suspend()` возвращает тип `void` или `bool`, то после приостановки текущей корутины управление безусловно возвращается вверх по стеку вызывающей стороне (тому потоку выполнения, который инициировал работу корутины). Это заставляет планировщик верхнего уровня заново искать готовую задачу в очереди и вызывать на ней метод `resume()`. При частых переключениях между мелкими корутинами такая схема приводит к накоплению стековых фреймов обычных функций C++ и может вызвать аппаратный `Stack Overflow`.

Для решения этой проблемы в C++20 был внедрен механизм **Symmetric Transfer (Симметричная передача управления)**.

> [!important] Суть Symmetric Transfer
> 
> Если метод `await_suspend()` возвращает объект `std::coroutine_handle<T>` _другой_ корутины, компилятор осуществляет прямой переход (tail-call jump) к исполнению целевой корутины. Стек текущего потока при этом не раздувается, поскольку фрейм текущей функции свертывается процессором.

Реализация авейтера с симметричной передачей для переключения контекста между двумя корутинами (аналог файберного yield):

```C++
struct YieldAwaiter {
    // Корутина всегда должна уснуть при вызове yield
    bool await_ready() const noexcept { return false; }

    // Метод принимает дескриптор уходящей в сон корутины
    std::coroutine_handle<> await_suspend(std::coroutine_handle<> current) noexcept {
        // Помещаем текущую корутину в глобальную очередь планировщика корутин
        CoroutineScheduler::Push(current);
        
        // Извлекаем следующую готовую корутину из очереди
        std::coroutine_handle<> next = CoroutineScheduler::Pop();
        
        if (next) {
            // Возвращаем дескриптор следующей корутины.
            // Компилятор выполнит Symmetric Transfer напрямую в её код!
            return next;
        }
        
        // Если готовых корутин нет, возвращаем специальный пустой handle,
        // управление вернется в поток ОС.
        return std::noop_coroutine(); 
    }

    void await_resume() noexcept {}
};
```

## 8. Ключевое слово `co_return` и жизненный цикл завершения

Классический оператор `return` заблокирован на уровне синтаксиса внутри корутин. Завершение работы бесстековой сопрограммы осуществляется через оператор `co_return`.

### Синтаксическая трансформация

1. Когда разработчик пишет `co_return <выражение>;`, компилятор транслирует эту команду во внутренний вызов метода объекта Promise:
    
    ```C++
    promise.return_value(<выражение>);
    goto final_suspend_label;
    ```
    
2. Если корутина не возвращает значение (`co_return;`) или выполнение естественным образом доходит до закрывающей фигурной скобки функции, компилятор подставляет вызов:
    
    ```C++
    promise.return_void();
    goto final_suspend_label;
    ```
    

### Точка `final_suspend`

После вызова `return_value()` или `return_void()` корутина переходит к финальной стадии своего существования и вызывает метод `promise.final_suspend()`.

> [!crit] Почему final_suspend обязан возвращать `std::suspend_always`?
> 
> Если метод `final_suspend()` вернет `std::suspend_never`, корутина немедленно автоматически уничтожит свой собственный `Coroutine Frame` в куче. Если к этому моменту вызывающий код (держатель объекта `Task`) еще не успел прочитать возвращенное значение из объекта Promise, он обратится к уничтоженной памяти, что приведет к неопределенному поведению (Use-After-Free).
> 
> Возврат `std::suspend_always` гарантирует, что фрейм корутины останется замороженным в памяти, позволяя безопасно извлечь результат выполнения операции. Уничтожение фрейма в этом случае становится обязанностью деструктора управляющего класса `Task` через вызов `handle.destroy()`.

## 9. Финальное сравнительное руководство (Concurrency Models Matrix)

|**Архитектурный критерий**|**Threads (Системные потоки)**|**Callbacks (Функции обратного вызова)**|**Fibers / Stackful Coroutines (Файберы)**|**Stackless Coroutines (Корутины C++20)**|
|---|---|---|---|---|
|**Расход памяти на единицу выполнения**|**Очень высокий** (от 2 до 8 МБ под жесткий стек операционной системы).|**Минимальный** (выделяется только замыкание под лямбду в куче).|**Средний** (фиксированный стек в user-space, обычно от 64 КБ на файбер).|**Минимальный** (размер фрейма точно оптимизируется компилятором, от десятков байт).|
|**Возможность приостановки на глубине стека**|**Да**, поток вытесняется или блокируется ядром ОС внутри любой вложенной функции.|**Нет**, функция должна полностью завершить выполнение и выйти.|**Да**, переключение контекста через ассемблер замораживает всю цепочку вызовов стека.|**Нет**, приостановка происходит строго на верхнем уровне через явный оператор `co_await`.|
|**Скорость переключения контекста**|**Низкая** (микросекунды, переход User-to-Kernel, инвалидация кэшей CPU и TLB).|**Максимальная** (наносекунды, эквивалентно обычному вызову указателя на функцию).|**Высокая** (наносекунды, подмена нескольких calлее-saved регистров без участия ядра).|**Высокая** (наносекунды, сохранение индекса состояния автомата без подмены стека).|
|**Линейность и читаемость исходного кода**|**Да**, код пишется последовательно.|**Нет**, структура кода разрушается из-за вложенности (_Callback Hell_).|**Да**, код выглядит абсолютно синхронным и последовательным.|**Да**, код линеен, но места приостановки явно размечены точками `co_await`.|
|**Фундаментальная уязвимость / Проблема модели**|Ограничение масштабирования (C10k) из-за нехватки RAM и оверхеда Kernel CPU.|Невозможность использования стандартных `try-catch`, циклов и декомпозиции функций.|**Риск Stack Overflow** при глубокой рекурсии из-за фиксированного малого размера стека.|**Фрагментация памяти** в куче при неэффективных аллокациях фреймов и «заражение» корутинами всего кода.|

**Ключевое архитектурное правило выбора:**

- Файберы оптимальны, когда требуется перевести огромную существующую синхронную кодовую базу (Legacy) на асинхронные рельсы без ее тотального переписывания. Достаточно подменить реализацию сокетов внутри системных библиотек, добавив туда контекст-свитч файберов.
    
- Бесстековые корутины C++20 представляют собой эталонный инструмент для написания новых высоконагруженных систем с нуля, поскольку обеспечивают максимальную производительность и минимальный след в памяти (Zero-cost abstraction) ценой необходимости явного проектирования асинхронных цепочек вызовов.
