
### Вариант 3. Улучшенный Спинлок (TTAS — Test and Test-and-Set) — ПРАВИЛЬНАЯ РЕАЛИЗАЦИЯ

Чтобы убрать шторм инвалидаций кэша, применяется стратегия: «Сначала просто посмотри обычным чтением, а когда освободится — пытайся записать». Обычный метод `.load()` требует кэш-линию только в состоянии _Shared_, что позволяет всем ядрам читать её локально без генерации трафика по шине.
```c++
class TTASSpinLock {
private:
    std::atomic<bool> state_{false};
public:
    void Lock() {
        while (true) {
            // Пытаемся захватить замок. Если вернулся false — мы выиграли.
            if (!state_.exchange(true)) {
                return;
            }
            // Если замок занят, крутимся в локальном кэше на обычном чтении (load)
            while (state_.load()) {
                // Spin-loop hint для процессора (опционально)
                #if defined(__x86_64__) || defined(_M_X64)
                _mm_pause(); // Предотвращает конвейерные спекуляции CPU
                #endif
            }
        }
    }
    void Unlock() {
        state_.store(false); 
    }
};
```

- **Сценарий работы TTAS:** Пока Поток А держит замок, Поток Б крутится во внутреннем цикле `while (state_.load())`. Кэш-линия лежит в L1-кэше ядра Потока Б в состоянии _Shared_. Никакого трафика по шине нет. Как только Поток А делает `Unlock` (`store(false)`), кэш-линия инвалидируется **ровно один раз**. Поток Б выходит из внутреннего цикла и делает попытку `exchange(true)`.

### Как написать идеальный мьютекс на фьютексе (Контекст курса PCP)

На лекциях по параллельному программированию подробно разбирается, почему простая булевская переменная (`true`/`false`) для фьютекса неэффективна. Если делать мьютекс на двух состояниях, то при каждом `unlock()` вам придется вызывать `FUTEX_WAKE` (делать системный вызов), даже если за замком никто не стоит и будить никого не надо.

Чтобы этого избежать, проектируют **мьютекс на 3 состояниях** (управляемых через одно 32-битное атомарное число):

- **`0` (Unlocked):** Мьютекс полностью свободен.
    
- **`1` (Locked, no waiters):** Мьютекс занят одним потоком, но за ним в очереди больше никто не стоит.
    
- **`2` (Locked, with waiters):** Мьютекс занят, и есть другие потоки, которые заблокировались и ушли спать в ядро.
    

**Как это оптимизирует работу?**

- **При захвате (Lock):** Поток пытается атомарно заменить `0` на `1`. Если получилось — мы захватили блокировку без сисколов. Если там уже была `1` или `2`, поток переводит состояние в `2` (указывая, что теперь точно есть ждущие) и вызывает `FUTEX_WAIT`.
    
- **При освобождении (Unlock):** Поток атомарно уменьшает значение или делает CAS.
    
    - Если состояние было `1` (никто не ждал), поток просто сбрасывает его в `0`. **Системный вызов `FUTEX_WAKE` не вызывается!** Мы сэкономили кучу времени.
        
    - Если состояние было `2` (кто-то спит в ядре), поток сбрасывает его в `0` и **обязательно вызывает `FUTEX_WAKE`**, чтобы разбудить спящего товарища.
        



## Раздел 2. Варианты реализации Mutex (Мьютексов) на базе Futex

Спинлоки сжигают 100% CPU, если критическая секция долгая. Для долгих задач нужен **Мьютекс**, который взаимодействует с операционной системой через системный вызов `futex` (Fast Userspace Mutex), усыпляя потоки в ядре ОС, когда замок занят.

### Вариант 1. Двухпозиционный Futex-мьютекс (0 и 1)

Самый простой вариант — использовать `std::atomic<int>` (где `0` — свободен, `1` — занят) и системные вызовы `futex_wait` и `futex_wake`.

C++

```
class NaiveFutexMutex {
private:
    std::atomic<int> state_{0}; // 0 - unlocked, 1 - locked
public:
    void Lock() {
        // Крутимся, пока не сможем атомарно перевести из 0 в 1
        while (state_.exchange(1) == 1) {
            // Уходим спать в ядро ОС. 
            // Сискол заснет только если значение в памяти реально равно 1.
            futex_wait(&state_, 1); 
        }
    }
    void Unlock() {
        state_.store(0);
        // ОШИБКА: Сискол wake вызывается ВСЕГДА!
        futex_wake(&state_, 1); 
    }
};
```

- **Как Фьютекс защищает от утерянного просыпания (Lost Wake-up Scenario):**
    
    _Сценарий гонки:_ Поток Б проверил `while`, замок занят (`state_ == 1`). Поток Б собирается вызвать `futex_wait`. Прямо перед системным вызовом контекст переключается. Поток А делает `Unlock()`: сбрасывает `state_` в `0` и делает `futex_wake`. Очередь пуста, сигнал улетает в никуда. Контекст возвращается к Потоку Б, и он вызывает `futex_wait(&state_, 1)`.
    
    _Почему нет вечного зависания:_ Системный вызов `futex_wait` внутри ядра атомарно перепроверяет: `if (*addr == val)`. Ядро видит, что в памяти по адресу `&state_` уже лежит `0`, а мы передали ожидаемое значение `1`. Проверка проваливается (`0 == 1` — ложь), ядро мгновенно возвращает управление Потоку Б с ошибкой `EWOULDBLOCK`, не укладывая его спать. Поток Б делает следующий круг цикла и забирает замок.
    
- **Критический недостаток этой реализации (Избыточные сисколы):**
    
    В методе `Unlock()` системный вызов `futex_wake` происходит **при каждом освобождении мьютекса**, даже если за замком никто не стоит (нет конкуренции). Переход в пространство ядра (Kernelspace) — дорогая операция (сотни наносекунд). Теряется весь смысл концепции "Fast Userspace", где в отсутствие конкуренции всё должно работать без сисколов.

### Вариант 2. Высокопроизводительный мьютекс на Трех Состояниях (0, 1, 2)

Чтобы полностью исключить системные вызовы на `Unlock()`, если конкуренции нет, состояние мьютекса расширяют до трех значений:

- **`0`**: Свободен (**Unlocked**).
    
- **`1`**: Занят, но за ним **никто не стоит** в очереди в ядре (**Locked, no waiters**).
    
- **`2`**: Занят, и в ядре гарантированно **есть ждущие потоки** (**Locked, HAS waiters**).
    

#### Пошаговые сценарии конкуренции (Важно для устного ответа)

- **Сценарий 1: Быстрый путь (Fast Path — Идеальный случай)**
    
    1. Поток А вызывает `Lock()`. Выполняется `compare_exchange_strong`: ожидаем `0`, хотим поставить `1`. Успешно! Поток А зашел в критическую секцию. (Сисколов к ОС: 0).
        
    2. Поток А завершил работу и вызывает `Unlock()`. Выполняется `state_.exchange(0)`. Назад возвращается старое значение `1`. Поток А понимает: _«Состояние оставалось единицей, пока я работал, ко мне никто не пришел»_. Поток А просто уходит. (Сисколов к ОС: 0).
        
- **Сценарий 2: Медленный путь (Slow Path — Появление конкурента)**
    
    1. Поток А удерживает мьютекс в состоянии `1`.
        
    2. Приходит Поток Б. Его стартовый CAS (`0 -> 1`) проваливается, так как там `1`.
        
    3. Поток Б понимает, что замок занят, и он обязан заставить Поток А сделать вейк при выходе. Поток Б атомарно (через CAS или exchange) переводит состояние мьютекса в `2` (`Locked, HAS waiters`).
        
    4. Поток Б вызывает `futex_wait(&state_, 2)`. Ядро проверяет память, видит там `2` и паркует поток спать.
        
    5. Поток А выходит из критической секции. Делает `state_.exchange(0)`. Ему возвращается старое значение `2`. Поток А понимает: _«За мной заняли очередь в ядре»_. Он обязан вызвать `futex_wake(&state_, 1)` для пробуждения Потока Б.
        
- **Сценарий 3: Накатывание (Феномен "Barging")**
    
    1. Поток А освободил замок, выставил `0` и дернул `futex_wake`. Поток Б (который спал в ядре) начинает процедуру пробуждения планировщиком ОС.
        
    2. Пока Поток Б медленно просыпается и переключает контекст (на это уходят сотни наносекунд), на исполнение процессора внезапно прорывается совершенно новый, горячий **Поток В**.
        
    3. Поток В вызывает `Lock()`, видит состояние `0`, успешно выполняет CAS (`0 -> 1`) и забирает замок без очереди!
        
    4. Наконец, окончательно просыпается Поток Б. Он выходит из `futex_wait` и идет на следующую итерацию цикла проверки. Он пытается выполнить захват, но видит, что в мьютексе уже лежит `1` (записанная Потоком В). Поток Б не паникует: он атомарно переводит состояние обратно из `1` в `2` и послушно уходит спать обратно в `futex_wait`.
        
        _Экзаменационный вывод:_ Мьютекс работает "нечестно" (нет гарантии FIFO), но такое поведение (**Barging**) осознанно допускается, так как оно максимизирует пропускную способность (Throughput) системы — процессорное ядро не простаивает ни одного такта, пока просыпающийся поток добирается до Userspace.

## Раздел 3. Анализ классических ошибок на примере выделенного счетчика ждущих (`waiters_count`)

Иногда вместо трех числовых состояний (`0, 1, 2`) программисты пытаются реализовать мьютекс, заведя два раздельных атомика: `std::atomic<bool> locked_` и счетчик `std::atomic<int> waiters_`. Это благодатная почва для критических багов.

### Ошибка 1. Race Condition и вечное засыпание (Lost Wake-up на счетчике)

Рассмотрим некорректный код:

C++

```
class BadMutex {
    std::atomic<bool> locked_{false};
    std::atomic<int> waiters_{0};
public:
    void Lock() {
        while (locked_.exchange(true)) { // (Шаг 1)
            waiters_.fetch_add(1);       // (Шаг 2)
            futex_wait(&locked_, true);  // (Шаг 3) Спим по адресу locked_
            waiters_.fetch_sub(1);
        }
    }
    void Unlock() {
        locked_.store(false);            // (Шаг 4)
        if (waiters_.load() > 0) {       // (Шаг 5)
            futex_wake(&locked_, 1);
        }
    }
};
```

- **Сценарий краха:**
    
    1. Поток А удерживает замок (`locked_ == true`, `waiters_ == 0`).
        
    2. Поток Б вызывает `Lock()`, на **Шаге 1** получает `true` (занято).
        
    3. Прямо перед тем, как Поток Б выполнит Шаг 2 (`fetch_add`), управление перехватывает Поток А.
        
    4. Поток А выполняет `Unlock()`: на **Шаге 4** делает `locked_.store(false)`. Затем на **Шаге 5** проверяет `waiters_.load()`. Так как Поток Б еще не успел инкрементировать счетчик, Поток А видит `0` и **НЕ вызывает** `futex_wake`.
        
    5. Контекст возвращается к Потоку Б. Он выполняет Шаг 2: `waiters_.fetch_add(1)`. Счетчик стал равен `1`.
        
    6. Поток Б выполняет Шаг 3: `futex_wait(&locked_, true)`. Поскольку ожидаемое значение (`true`) не совпадает с фактическим (`false`), ядро не укладывает его спать, а возвращает `EWOULDBLOCK`. Поток Б выходит в цикл `while`, делает `exchange(true)` и забирает замок.
        
- **А где же баг?** Поменяем сценарий: Поток Б успел сделать `fetch_add(1)`, но застрял _между_ `fetch_add(1)` и вызовом `futex_wait`. Поток А заходит в `Unlock()`, видит `waiters_ == 1`, вызывает `futex_wake` (сигнал уходит в никуда, так как Поток Б еще не в ядре!). Затем Поток Б заходит в `futex_wait(&locked_, true)`. Если в этот момент прилетит Поток В, выставит `locked_ = true`, то проверка в ядре для Потока Б совпадет (`true == true`), и Поток Б **уснет в ядре навсегда**, несмотря на то, что Поток А уже отпустил замок!

### Ошибка 2. Data Race при чтении счетчика вне блокировки (Начало 6 недели)

В многопоточных примитивах синхронизации часто совершают ошибку, выполняя инкремент разделяемой переменной под внутренним локом, а чтение — вне лока без атомарных гарантий.

- **Суть проблемы:** Как подчеркивается на лекциях, если запись (модификация) переменной происходит под локом мьютекса, а чтение этой же переменной выполняется в другом потоке конкурентно и без синхронизации (не под мьютексом и не через упорядоченные атомарные операции), возникает **Data Race**.
    
- **Экзаменационное исправление:** Любое разделяемое состояние (счетчик ждущих, флаги завершения) обязано управляться через атомики `std::atomic` с четким пониманием барьеров памяти (`memory_order`), либо доступ к нему должен быть строго линеаризован общим мьютексом.

## Чек-лист для экзамена: Ключевые вопросы по 4-6 неделям

1. **В чем аппаратная разница между `exchange` и `load` в цикле Спинлока?**
    
    - `exchange` — это запись (RMW), требующая эксклюзивного владения кэш-линией (инвалидация кэшей других ядер при каждом вызове). `load` — это чтение, позволяющее кэш-линии мирно находиться в состоянии _Shared_ на всех ядрах без генерации системного трафика по шине.
        
2. **Зачем нужен макрос/инструкция `_mm_pause()` внутри Spin-loop?**
    
    - Она сообщает процессору, что выполняется цикл активного ожидания. Это оптимизирует энергопотребление ядра и предотвращает дорогостоящую очистку конвейера спекулятивных инструкций процессора при выходе из цикла.
        
3. **Какую роль играет аргумент `val` в системном вызове `futex_wait(addr, val)`?**
    
    - Он критически необходим для предотвращения race condition "Lost Wake-up". Ядро гарантирует атомарную проверку: если значение по адресу `addr` изменилось и больше не равно `val`, поток не засыпает, защищаясь от утери отправленного ранее сигнала пробуждения.
        
4. **Почему трехрежимный мьютекс быстрее двухрежимного?**
    
    - Потому что состояние `1` (Locked, no waiters) позволяет методу `Unlock()` обойтись обычной атомарной операцией в Userspace без вызова тяжелого системного вызова `futex_wake` к операционной системе. Сискол делается только тогда, когда состояние было равно `2`.
