==Опр.== **Spinlock** - это простейший примитив [[Mutex|mutex]], работающий на принципе активного ожидания (Busy Waiting).

**Принцип работы:** Поток, пытающийся захватить объект, крутится в бесконечном цикле `while`, постоянно проверяя значение разделяемой переменной-флага (например, занято/свободно).

## Реализации Spinlock
### 1. Spinlock на обычном флаге

```c++
class SpinLock {
public:
	void Lock() {
	    while (flag) { /* busy wait */ }
	    flag = true;
	}
	
	void Unlock() {
	    flag = false;
	}

private:
	bool flag = false; 
};
```

Данная реализация имеет критические ошибки и не работает по нескольким причинам:

1) **Оптимизация компилятора** 
	Компилятор видит, что внутри цикла `while (flag) {}` значение `flag` не меняется текущим потоком. Он имеет право закешировать это значение в регистр процессора один раз. В итоге, даже если другой поток изменит `flag` в памяти, ждущий поток об этом никогда не узнает и зависнет в вечном цикле.
    
2) **Data Race**
	Операции проверки `while(flag)` и последующей установки `flag = true` не выполняются неделимо. Два потока могут одновременно увидеть, что `flag == false`, оба выйдут из цикла и оба установят `flag = true`, зайдя в критическую секцию одновременно. Это нарушает инвариант взаимного исключения и приводит к UB.
    
3) **Store Buffer** 
	Каждое ядро процессора имеет сверхбыстрый локальный буфер - **Store Buffer**. Когда поток выполняет операцию записи (`store`), данные физически попадают сначала в этот буфер, а не в L1-кэш или общую оперативную память. Другие ядра увидят изменения только тогда, когда Store Buffer будет сброшен в общую систему кэш-памяти, и сразу несколько потоков одновременно увидят `flag = false` и зайдут в критическую секцию.

### 2. [[Atomic|Атомарный]] Spinlock

Исправляя предыдущие проблемы, мы переходим к использованию atomic и RMW-операций. Метод `.exchange(true)` атомарно записывает `true` и возвращает **старое** значение переменной.
```c++
class SpinLock {
private:
    std::atomic<bool> state_{false};
    
public:
    void Lock() {
        // Атомарно пытаемся установить true. 
        while (state_.exchange(true)) {
            // busy wait
        }
    }
    
    void Unlock() {
        state_.store(false); // Атомарно освобождаем
    }
};
```

#### Как работает?
1. Поток А вызывает `Lock()`. `state_.exchange(true)` видит старое значение `false`, записывает `true`, возвращает `false`. Цикл завершается. Поток А в критической секции.
2. Приходит Поток Б. Вызывает `Lock()`. `exchange(true)` видит текущее значение `true`, записывает `true` и возвращает `true`. Поток Б остается крутиться в цикле.

#### В чём недостаток?
Каждая RMW-инструкция на аппаратном уровне трактуется процессором как операция записи. Согласно протоколам когерентности кэшей *(например, MESI)*, для выполнения записи ядро процессора обязано запросить эксклюзивное владение кэш-линией, содержащей `state_`, и отправить сигнал инвалидации во все остальные ядра. Если 10 потоков будут непрерывно вызывать `exchange(true)`, кэш-линия будет с огромной скоростью «прыгать» от одного ядра к другому по межпроцессорной шине. Этот «шторм» полностью забивает пропускную способность шины памяти, замедляя работу всей системы, включая Поток А, который сейчас находится в критической секции и пытается делать полезную работу.

### 3. TTAS Spinlock

Чтобы убрать шторм инвалидаций кэша, применяется стратегия TTAS (Test and Test-and-Set): «Сначала просто посмотри обычным чтением, а когда освободится - пытайся записать». Обычный метод `.load()` требует кэш-линию только в состоянии _Shared_, что позволяет всем ядрам читать её локально без генерации трафика по шине.
```c++
class TTASSpinLock {
private:
    std::atomic<bool> state_{false};
    
public:
    void Lock() {
        while (true) {
            // Пытаемся захватить замок. Если вернулся false - мы выиграли.
            if (!state_.exchange(true)) {
                return;
            }
            // Если замок занят, крутимся в локальном кэше на обычном чтении
            while (state_.load()) {
                // Spin-loop hint для процессора (опционально)
                #if defined(__x86_64__) || defined(_M_X64)
                _mm_pause(); // Предотвращает конвейерные спекуляции CPU
                #endif
            }
        }
    }
    
    void Unlock() {
        state_.store(false); 
    }
};
```

#### Как работает?
Пока Поток А держит замок, Поток Б крутится во внутреннем цикле `while (state_.load())`. Кэш-линия лежит в L1-кэше ядра Потока Б в состоянии _Shared_. Никакого трафика по шине нет. Как только Поток А делает `Unlock`, кэш-линия инвалидируется **ровно один раз**. Поток Б выходит из внутреннего цикла и делает попытку `exchange(true)`.

### +ы и -ы Spinlock

> [!success] Преимущество
> Если критическая секция выполняется очень быстро *(пару тактов процессора)*, spinlock идеален. Поток не тратит время на переключение в контекст операционной системы. Реакция на освобождение замка - мгновенная.

> [!failure] Недостатки
> 1. **Расход CPU:** Поток сжигает 100% времени одного процессорного ядра на абсолютно холостой цикл.
> 2. **Проблема вытеснения:** Если поток, который уже удерживает spinlock, будет вытеснен планировщиком ОС, все остальные потоки, ждущие этот spinlock, будут безумно крутиться в цикле, тратя впустую миллионы тактов CPU, пока планировщик не вернёт контекст первому потоку.
