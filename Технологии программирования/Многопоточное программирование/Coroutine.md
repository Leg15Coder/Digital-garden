==Опр.== **Корутина (Coroutine, сопрограмма)** - это специальный вид функций, которые могут приостанавливать своё выполнение в определённой точке, сохранять текущее состояние и возобновляться позже с места остановки. Это основной инструмент для создания асинхронного и конкурентного кода, позволяющий выполнять множество задач без блокировки основного потока.

Существует два принципиально разных подхода к реализации сопрограмм на уровне архитектуры вычислительных систем:

1) **Stackful корутины ([[Fiber]])**
	- **Механизм сохранения состояния:** Каждая сопрограмма имеет выделенный стек в адресном пространстве процесса. Вся информация о локальных переменных и цепочке вызовов сохраняется на этом стеке.
	- **Глубина приостановки:** Корутина может быть приостановлена **на любой глубине вложенности вызовов**. Если функция `MainFiberFunc()` вызывает глубоко вложенную библиотечную функцию `ПосчитатьМетрики()`, а та вызывает сетевой `Read()`, внутри которого вызывается `yield()`, планировщик успешно заморозит _весь_ стек вызовов.
	- **Требования к языку/компилятору:** Минимальные. Компилятор может вообще ничего не знать про существование корутин. Вся логика реализуется библиотечно за счёт ассемблерных манипуляций с регистром `RSP` процессора.

2)  **Stackless корутины**
- **Механизм сохранения состояния:** Корутина не имеет собственного стека. Локальные переменные и параметры, необходимые для возобновления работы функции, упаковываются компилятором в компактный объект - **Coroutine Frame**, размещаемый в куче.
- **Глубина приостановки:** Корутина может быть приостановлена **исключительно на верхнем уровне своего тела** с помощью явного ключевого слова (`co_await`). Если корутина вызывает обычную функцию `DoWork()`, внутри `DoWork()` приостановить корутину невозможно. Чтобы приостановка была возможна на глубине, вся цепочка вызовов обязана быть корутинами (`Task<T>`) и вызываться через явный синтаксис `co_await DoWorkAsync()`.
- **Требования к языку/компилятору:** Требуется глубокая интеграция на уровне спецификации языка и компилятора. Компилятор полностью переписывает исходный код функции, превращая её в [[Детерминированный конечный автомат|ДКА]].

### Сходства и различия coroutine и fiber

**Сходства:**
- Обе модели предоставляют абстракцию кооперативной многозадачности.
- Позволяют писать асинхронный код ввода-вывода в линейном, интуитивно понятном стиле, избавляя от Callback Hell.

**Различия:**
- Fiber требует под свой стек фиксированный объём, даже если глубина вызовов составляет всего один фрейм. Stackless корутина расходует ровно столько байт в куче, сколько весят её локальные переменные.
- В корутинах точки приостановки всегда видны в коде (ключевое слово `co_await`). В fibers вызов `yield()` или блокировка на мьютексе скрыты внутри вызываемых функций.

### Построение ДКА по корутине

Когда компилятор встречает функцию, содержащую `co_await`, `co_yield` или `co_return`, он радикально трансформирует её структуру. Компилятор превращает тело функции в класс, реализующий конечный автомат. Каждая точка `co_await` делит функцию на отдельные сегменты выполнения. Рассмотрим исходную корутину:
```C++
Task MyCoroutine(int a) {
    int x = a + 5;
    co_await SomeAsyncOperation();
    int y = x * 2;
    co_return y;
}
```

Компилятор преобразует этот код в структуру следующего вида *(псевдокод)*:
```C++
struct MyCoroutine_Frame {
    // Внутреннее состояние конечного автомата
    int __state = 0; 
    
    // Объект Promise, управляющий корутиной
    typename Task::promise_type __promise; 
    
    // Входные параметры функции
    int a; 
    
    // Локальные переменные, которые должны пережить приостановку корутины
    int x; 
    int y;
    
    // Скрытый объект-awaiter для текущей операции
    decltype(SomeAsyncOperation()) __awaiter; 
};

// Функция, вызываемая при возобновлении корутины
void MyCoroutine_Resume(MyCoroutine_Frame* __frame) {
    switch (__frame->__state) {
        case 0:
            goto __state_0;
        case 1:
            goto __state_1;
    }

__state_0:
    __frame->x = __frame->a + 5;
    __frame->__awaiter = SomeAsyncOperation();
    
    if (!__frame->__awaiter.await_ready()) {
        __frame->__state = 1; // Запоминаем точку приостановки
        // Передаём дескриптор корутины авейтеру для подписки на событие
        __frame->__awaiter.await_suspend(std::coroutine_handle<Task::promise_type>::from_address(__frame));
        return; // Возвращаем управление вызывающему потоку
    }

__state_1:
    // Корутина проснулась здесь
    __frame->y = __frame->__awaiter.await_resume() * 2;
    __frame->__promise.return_value(__frame->y);
    // Финальная стадия жизненного цикла корутины
}
```

> [!info] Локальные переменные на стеке и в куче
> 
> Переменные корутины, время жизни которых не пересекает точку приостановки `co_await`, компилятор может продолжать размещать на обычном аппаратном стеке текущего потока в момент вызова `resume()`. Однако те переменные, значения которых необходимы после возобновления работы корутины, гарантированно переносятся во фрейм корутины в кучу.

> [!tip] **Оптимизация HALO (Heap Allocation Elimination)**
> Если компилятор на этапе статического анализа видит, что время жизни корутины строго ограничено временем жизни вызывающей функции (то есть корутина гарантированно завершится до выхода из вызывающего скоупа), он имеет право применить оптимизацию HALO и выделить память под Coroutine Frame на обычном стеке вызывающей функции, полностью исключив обращение к куче.
