==Опр.== **Корутина (Coroutine, сопрограмма)** - это специальный вид функций, которые могут приостанавливать своё выполнение в определённой точке, сохранять текущее состояние и возобновляться позже с места остановки. Это основной инструмент для создания асинхронного и конкурентного кода, позволяющий выполнять множество задач без блокировки основного потока.

Существует два принципиально разных подхода к реализации сопрограмм на уровне архитектуры вычислительных систем:

1) **Stackful корутины ([[Fiber]])**
	- **Механизм сохранения состояния:** Каждая сопрограмма имеет выделенный стек в адресном пространстве процесса. Вся информация о локальных переменных и цепочке вызовов сохраняется на этом стеке.
	- **Глубина приостановки:** Корутина может быть приостановлена **на любой глубине вложенности вызовов**. Если функция `MainFiberFunc()` вызывает глубоко вложенную библиотечную функцию `ПосчитатьМетрики()`, а та вызывает сетевой `Read()`, внутри которого вызывается `yield()`, планировщик успешно заморозит _весь_ стек вызовов.
	- **Требования к языку/компилятору:** Минимальные. Компилятор может вообще ничего не знать про существование корутин. Вся логика реализуется библиотечно за счёт ассемблерных манипуляций с регистром `RSP` процессора.

2)  **Stackless корутины**
- **Механизм сохранения состояния:** Корутина не имеет собственного стека. Локальные переменные и параметры, необходимые для возобновления работы функции, упаковываются компилятором в компактный объект - **Coroutine Frame**, размещаемый в куче.
- **Глубина приостановки:** Корутина может быть приостановлена **исключительно на верхнем уровне своего тела** с помощью явного ключевого слова (`co_await`). Если корутина вызывает обычную функцию `DoWork()`, внутри `DoWork()` приостановить корутину невозможно. Чтобы приостановка была возможна на глубине, вся цепочка вызовов обязана быть корутинами (`Task<T>`) и вызываться через явный синтаксис `co_await DoWorkAsync()`.
- **Требования к языку/компилятору:** Требуется глубокая интеграция на уровне спецификации языка и компилятора. Компилятор полностью переписывает исходный код функции, превращая её в [[Детерминированный конечный автомат|ДКА]].

### Сходства и различия coroutine и fiber

**Сходства:**
- Обе модели предоставляют абстракцию кооперативной многозадачности.
- Позволяют писать асинхронный код ввода-вывода в линейном, интуитивно понятном стиле, избавляя от Callback Hell.

**Различия:**
- Fiber требует под свой стек фиксированный объём, даже если глубина вызовов составляет всего один фрейм. Stackless корутина расходует ровно столько байт в куче, сколько весят её локальные переменные.
- В корутинах точки приостановки всегда видны в коде (ключевое слово `co_await`). В fibers вызов `yield()` или блокировка на мьютексе скрыты внутри вызываемых функций.

### Построение ДКА по корутине

Когда компилятор встречает функцию, содержащую `co_await`, `co_yield` или `co_return`, он радикально трансформирует её структуру. Компилятор превращает тело функции в класс, реализующий конечный автомат. Каждая точка `co_await` делит функцию на отдельные сегменты выполнения. Рассмотрим исходную корутину:
```C++
Task MyCoroutine(int a) {
    int x = a + 5;
    co_await SomeAsyncOperation();
    int y = x * 2;
    co_return y;
}
```

Компилятор преобразует этот код в структуру следующего вида *(псевдокод)*:
```C++
struct MyCoroutine_Frame {
    // Внутреннее состояние конечного автомата
    int __state = 0; 
    
    // Объект Promise, управляющий корутиной
    typename Task::promise_type __promise; 
    
    // Входные параметры функции
    int a; 
    
    // Локальные переменные, которые должны пережить приостановку корутины
    int x; 
    int y;
    
    // Скрытый объект-awaiter для текущей операции
    decltype(SomeAsyncOperation()) __awaiter; 
};

// Функция, вызываемая при возобновлении корутины
void MyCoroutine_Resume(MyCoroutine_Frame* __frame) {
    switch (__frame->__state) {
        case 0:
            goto __state_0;
        case 1:
            goto __state_1;
    }

__state_0:
    __frame->x = __frame->a + 5;
    __frame->__awaiter = SomeAsyncOperation();
    
    if (!__frame->__awaiter.await_ready()) {
        __frame->__state = 1; // Запоминаем точку приостановки
        // Передаём дескриптор корутины авейтеру для подписки на событие
        __frame->__awaiter.await_suspend(std::coroutine_handle<Task::promise_type>::from_address(__frame));
        return; // Возвращаем управление вызывающему потоку
    }

__state_1:
    // Корутина проснулась здесь
    __frame->y = __frame->__awaiter.await_resume() * 2;
    __frame->__promise.return_value(__frame->y);
    // Финальная стадия жизненного цикла корутины
}
```

> [!info] Локальные переменные на стеке и в куче
> 
> Переменные корутины, время жизни которых не пересекает точку приостановки `co_await`, компилятор может продолжать размещать на обычном аппаратном стеке текущего потока в момент вызова `resume()`. Однако те переменные, значения которых необходимы после возобновления работы корутины, гарантированно переносятся во фрейм корутины в кучу.

> [!tip] **Оптимизация HALO (Heap Allocation Elimination)**
> Если компилятор на этапе статического анализа видит, что время жизни корутины строго ограничено временем жизни вызывающей функции (то есть корутина гарантированно завершится до выхода из вызывающего скоупа), он имеет право применить оптимизацию HALO и выделить память под Coroutine Frame на обычном стеке вызывающей функции, полностью исключив обращение к куче.

## Внутреннее устройство stackless coroutines

Управление stackless coroutine распределено между тремя компонентами:
![[Анатомия_бесстековой_корутины_C++20.png]]

1. **Coroutine Frame:** Внутренний блок памяти в куче, содержащий Promise, аргументы, локальные переменные и стейт-индекс.
2. **Promise Object:** Создаётся внутри фрейма. Это интерфейс, через который автор библиотеки настраивает поведение корутины. Обязан содержать методы `initial_suspend`, `final_suspend`, `get_return_object`, `unhandled_exception`, `return_value` или `return_void`.
3. **Coroutine Handle (`std::coroutine_handle<>`\):** Не владеющий указатель на фрейм, доступный пользователю.

Реализация класса `Task` с объектом Promise:
```C++
class Task {
public:
    struct promise_type;
    using handle_type = std::coroutine_handle<promise_type>;

    struct promise_type {
        // Вызывается компилятором первым делом для конструирования внешнего Task
        Task get_return_object() {
            return Task(handle_type::from_promise(*this));
        }

        // Задаём поведение при старте: приостановить выполнение (Lazy evaluation)
        std::suspend_always initial_suspend() { return {}; }
        // Задаём поведение при завершении: приостановить, чтобы сохранить promise живым
        std::suspend_always final_suspend() noexcept { return {}; }

        // Сохраняем возвращаемое значение корутины
        void return_value(int value) {
            value_ = value;
        }

        // Перехватываем исключения, выброшенные в корутине
        void unhandled_exception() {
            exception_ = std::current_exception();
        }

        int value_ = 0;
        std::exception_ptr exception_;
    };

    Task(handle_type handle) : handle_(handle) {}
    
    ~Task() {
        if (handle_) handle_.destroy(); // Ручное освобождение фрейма в куче
    }

    void Resume() {
        if (handle_ && !handle_.done()) {
            handle_.resume(); // Передаём процессор корутине
        }
    }

    int GetResult() {
        if (handle_.promise().exception_) {
            std::rethrow_exception(handle_.promise().exception_);
        }
        return handle_.promise().value_;
    }

private:
    handle_type handle_;
};
```

### Симметричная передача управления

Если метод `await_suspend()` возвращает тип `void` или `bool`, то после приостановки текущей корутины управление безусловно возвращается вверх по стеку вызывающей стороне (тому потоку выполнения, который инициировал работу корутины). Это заставляет планировщик верхнего уровня заново искать готовую задачу в очереди и вызывать на ней метод `resume()`. При частых переключениях между мелкими корутинами такая схема приводит к накоплению стековых фреймов обычных функций C++ и может вызвать аппаратный `Stack Overflow`. Для решения этой проблемы в C++20 был внедрен механизм **Symmetric Transfer (Симметричная передача управления)**.

> [!important] Суть Symmetric Transfer
> 
> Если метод `await_suspend()` возвращает объект `std::coroutine_handle<T>` _другой_ корутины, компилятор осуществляет прямой переход (tail-call jump) к исполнению целевой корутины. Стек текущего потока при этом не раздувается, поскольку фрейм текущей функции свёртывается процессором.

Реализация awaiter с симметричной передачей для переключения контекста между двумя корутинами *(аналог yield)*:

```C++
struct YieldAwaiter {
    // Корутина всегда должна уснуть при вызове yield
    bool await_ready() const noexcept { return false; }

    // Метод принимает дескриптор уходящей в сон корутины
    std::coroutine_handle<> await_suspend(std::coroutine_handle<> current) noexcept {
        // Помещаем текущую корутину в глобальную очередь планировщика корутин
        CoroutineScheduler::Push(current);
        // Извлекаем следующую готовую корутину из очереди
        std::coroutine_handle<> next = CoroutineScheduler::Pop();
        
        if (next) {
            // Возвращаем дескриптор следующей корутины.
            // Компилятор выполнит Symmetric Transfer напрямую в её код!
            return next;
        }
        
        // Если готовых корутин нет, возвращаем специальный пустой handle,
        // управление вернётся в поток ОС.
        return std::noop_coroutine(); 
    }

    void await_resume() noexcept {}
};
```

### Разбор ключевого слова `co_await`

Когда в теле корутины вызывается выражение `co_await expr;`, компилятор C++20 разворачивает его в цепочку вызовов трех методов специального объекта - **Awaiter**. Псевдокод развёртывания выглядит следующим образом:
```C++
auto&& awaiter = get_awaiter(expr); // Получаем объект-awaiter

if (!awaiter.await_ready()) {
    // Корутина фиксирует текущую точку и переходит в состояние Suspended
    // Вызов suspend с передачей дескриптора текущей корутины
    awaiter.await_suspend(current_coroutine_handle); 
    // Управление возвращается тому потоку, который вызвал handle.resume()
}
// Вызывается при пробуждении. Результат становится результатом всего выражения co_await
auto result = awaiter.await_resume(); 
```

- **`bool await_ready()`:** Позволяет избежать приостановки, если асинхронная операция уже завершилась к моменту вызова. Например, если мы ждём данные из сокета, а они уже находятся в буфере ядра. Если метод возвращает `true`, корутина продолжает выполнение без засыпания.
- **`await_suspend(std::coroutine_handle<> handle\)`:** Метод вызывается, когда корутина уже гарантированно приостановлена. Внутри этого метода разработчик регистрирует `handle` в структурах данных Event Loop *(например, связывает с системным дескриптором `epoll`)*.
- **`await_resume()`:** Возвращает результат выполнения асинхронной операции. Если асинхронная операция завершилась ошибкой, именно внутри `await_resume()` генерируется исключение C++, которое легитимно ловится внешним блоком `try-catch` в теле корутины.

### Ключевое слово `co_return`

Классический оператор `return` заблокирован на уровне синтаксиса внутри корутин. Завершение работы stackless сопрограммы осуществляется через оператор `co_return`.

1. Когда разработчик пишет `co_return expr;`, компилятор транслирует эту команду во внутренний вызов метода объекта Promise:
    ```C++
    promise.return_value(expr);
    goto final_suspend_label;
    ```
2. Если корутина не возвращает значение (`co_return;`) или выполнение естественным образом доходит до закрывающей фигурной скобки функции, компилятор подставляет вызов:
    ```C++
    promise.return_void();
    goto final_suspend_label;
    ```

После вызова `return_value()` или `return_void()` корутина переходит к финальной стадии своего существования и вызывает метод `promise.final_suspend()`.

> [!crit] Почему `final_suspend` обязан возвращать `std::suspend_always`?
> 
> Если метод `final_suspend()` вернёт `std::suspend_never`, корутина немедленно автоматически уничтожит свой собственный `Coroutine Frame` в куче. Если к этому моменту вызывающий код (держатель объекта `Task`) ещё не успел прочитать возвращённое значение из объекта Promise, он обратится к уничтоженной памяти, что приведёт к UB.
> 
> Возврат `std::suspend_always` гарантирует, что фрейм корутины останется замороженным в памяти, позволяя безопасно извлечь результат выполнения операции. Уничтожение фрейма в этом случае становится обязанностью деструктора управляющего класса `Task` через вызов `handle.destroy()`.

## Сравнение методов управления потоками

| **Архитектурный критерий**                    | **System Threads**                                                                 | **Callbacks**                                                                       | **Fibers**                                                                               | **Stackless Coroutines**                                                                                 |
| --------------------------------------------- | ---------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **Расход памяти на единицу выполнения**       | **Очень высокий** (от 2 до 8 МБ под жёсткий стек ОС).                              | **Минимальный** (выделяется только замыкание под лямбду в куче).                    | **Средний** (фиксированный стек в user-space, обычно от 64 КБ на fiber).                 | **Минимальный** (размер фрейма точно оптимизируется компилятором, от десятков байт).                     |
| **Возможность приостановки на глубине стека** | **Да**, поток вытесняется или блокируется ядром ОС внутри любой вложенной функции. | **Нет**, функция должна полностью завершить выполнение и выйти.                     | **Да**, переключение контекста через ассемблер замораживает всю цепочку вызовов стека.   | **Нет**, приостановка происходит строго на верхнем уровне через явный оператор `co_await`.               |
| **Скорость переключения контекста**           | **Низкая** (микросекунды, переход User-to-Kernel, инвалидация кэшей CPU и TLB).    | **Максимальная** (наносекунды, эквивалентно обычному вызову указателя на функцию).  | **Высокая** (наносекунды, подмена нескольких calлее-saved регистров без участия ядра).   | **Высокая** (наносекунды, сохранение индекса состояния автомата без подмены стека).                      |
| **Линейность и читаемость исходного кода**    | **Да**, код пишется последовательно.                                               | **Нет**, структура кода разрушается из-за вложенности (_Callback Hell_).            | **Да**, код выглядит абсолютно синхронным и последовательным.                            | **Да**, код линеен, но места приостановки явно размечены точками `co_await`.                             |
| **Фундаментальная уязвимость**                | Ограничение масштабирования (C10k) из-за нехватки RAM и перегрузки Kernel CPU.     | Невозможность использования стандартных `try-catch`, циклов и декомпозиции функций. | **Риск Stack Overflow** при глубокой рекурсии из-за фиксированного малого размера стека. | **Фрагментация памяти** в куче при неэффективных аллокациях фреймов и «заражение» корутинами всего кода. |

#### Ключевое архитектурное правило выбора:

- Fibers оптимальны, когда требуется перевести огромную существующую синхронную кодовую базу (Legacy) на асинхронные рельсы без её тотального переписывания. Достаточно подменить реализацию сокетов внутри системных библиотек, добавив туда контекст-свитч fibers.
- Stackless coroutines представляют собой эталонный инструмент для написания новых высоконагруженных систем с нуля, поскольку обеспечивают максимальную производительность и минимальный след в памяти ценой необходимости явного проектирования асинхронных цепочек вызовов.
