## Параллелизм vs Конкурентность

Начиная изучение многопоточности, важно различать два понятия, которые часто путают:
* **Параллельность (Parallelism):** Когда у вас есть несколько вычислительных задач и несколько ядер процессора. Задачи выполняются физически одновременно. 
    * *Аналогия:* Много мисок с едой, и каждый котик ест из своей миски (едят одновременно)
* **Конкурентность (Concurrency):** Способ организации доступа к разделяемому ресурсу или обработки множества запросов. Это не обязательно означает одновременное выполнение, но подразумевает продвижение по нескольким задачам сразу.
    * *Аналогия:* Одна миска, в которую пытаются залезть все котики сразу. Возникают проблемы синхронизации и порядка доступа.

**Зачем это серверу?** Современные высоконагруженные системы сталкиваются с проблемой **C10k** (обработка 10 000 соединений одновременно). Обычного создания потока на каждый запрос недостаточно - это неэффективно по памяти и времени на переключение контекста.

**Что такое Протокол?** Это договоренность (набор фраз) о том, как общаться, чтобы стороны понимали друг друга однозначно.

## Сокет

==Опр.== **Сокет** - это программный интерфейс (абстракция) для сетевых соединений, реализованный в ядре ОС. В Linux сокет представляется как **файловый дескриптор**. 

Для создания простейшего сервера на C++ нужно последовательно вызвать несколько системных функций:

1. **`socket()`**: Создаёт точку входа. Параметры: `AF_INET` (IPv4) и `SOCK_STREAM` (TCP).
2. **`bind()`**: Регистрация номера. Привязывает сокет к конкретному IP-адресу и **порту** (число от 0 до 65535) 
    *   *Нюанс:* Порты < 1024 зарезервированы для системы (нужны права `root`).
3. **`listen()`**: Переводит сокет в состояние прослушивания (по умолчанию сокет создан, но не готов принимать клиентов). Параметр **backlog** определяет размер очереди клиентов, которые могут «висеть на линии», пока сервер их не принял.
4. **`accept()`**: Блокирующий вызов. Сервер засыпает, пока не придёт клиент. При подключении `accept` возвращает **новый файловый дескриптор** специально для общения с этим клиентом.

> [!IMPORTANT] Важное различие
> На сервере всегда есть **слушающий сокет** (только принимает звонки) и **клиентский сокет** (создаётся для каждого конкретного разговора).

### Чем определяется TCP-соединение?
TCP-соединение однозначно определяется четверкой значений (кортежем):
1. IP-адрес клиента.
2. Порт клиента (назначается ОС случайно при подключении).
3. IP-адрес сервера.
4. Порт сервера.
Благодаря этому сервер может поддерживать множество соединений на одном и том же порту.

### Проблемы однопоточной модели
В простейшем сервере обработка идёт в цикле: принял клиента -> прочитал -> ответил -> закрыл.
* **Проблема:** Если один клиент подключился и молчит, вызов `read` заблокирует весь сервер. Другие клиенты будут стоять в очереди ядра (backlog) и не получат ответа, пока первый не договорит.
* **Решение:** Выносить обработку каждого клиента в отдельный поток (`std::thread)`, а затем использовать пулы потоков и неблокирующий ввод-вывод.

### Проблема C10k

Несмотря на простоту, подход «один поток на одного клиента» имеет серьёзные ограничения, известные как проблема C10k:
- **Затраты на создание потоков:** Каждый поток требует ресурсов ОС и выделения памяти под стек (обычно около 1-8 МБ). 10,000 потоков могут исчерпать оперативную память.
- **Переключение контекста (Context Switch):** Когда потоков слишком много, процессор тратит больше времени на переключение между ними, чем на полезную работу.
- **Масштабируемость:** ОС плохо справляется с управлением десятками тысяч конкурирующих потоков.

#### Инструментарий для новичка
* **`netstat -tan`**: Посмотреть список открытых портов и состояние соединений (`LISTEN`, `ESTABLISHED`).
* **`Netcat (nc)`**: Универсальный швейцарский нож для отправки сырых байтов по TCP/UDP.
* **`telnet`**: Простая утилита для проверки доступности порта.
* **`curl -v`**: Проверка HTTP-серверов с отображением заголовков.
* **`/proc/[pid]/fd`**: Директория в Linux, где можно увидеть все открытые сокеты вашего процесса.
