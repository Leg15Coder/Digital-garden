# <h1 id="p1"><center>Жизненный цикл TCP-сервера</center></h1>

## Параллелизм vs Конкурентность

Начиная изучение многопоточности, важно различать два понятия, которые часто путают:
* **Параллельность (Parallelism):** Когда у вас есть несколько вычислительных задач и несколько ядер процессора. Задачи выполняются физически одновременно. 
    * *Аналогия:* Много мисок с едой, и каждый котик ест из своей миски (едят одновременно)
* **Конкурентность (Concurrency):** Способ организации доступа к разделяемому ресурсу или обработки множества запросов. Это не обязательно означает одновременное выполнение, но подразумевает продвижение по нескольким задачам сразу.
    * *Аналогия:* Одна миска, в которую пытаются залезть все котики сразу. Возникают проблемы синхронизации и порядка доступа.

![[Gemini_Generated_Image_i0or5mi0or5mi0or.png]]

**Зачем это серверу?** Современные высоконагруженные системы сталкиваются с проблемой **C10k** (обработка 10 000 соединений одновременно). Обычного создания потока на каждый запрос недостаточно - это неэффективно по памяти и времени на переключение контекста.

**Что такое Протокол?** Это договоренность (набор фраз) о том, как общаться, чтобы стороны понимали друг друга однозначно.

## Сокет

==Опр.== **Сокет** - это программный интерфейс (абстракция) для сетевых соединений, реализованный в ядре ОС. В Linux сокет представляется как **файловый дескриптор**. 

Для создания простейшего сервера на C++ нужно последовательно вызвать несколько системных функций:

1. **`socket()`**: Создаёт точку входа. Параметры: `AF_INET` (IPv4) и `SOCK_STREAM` (TCP).
2. **`bind()`**: Регистрация номера. Привязывает сокет к конкретному IP-адресу и **порту** (число от 0 до 65535) 
    *   *Нюанс:* Порты < 1024 зарезервированы для системы (нужны права `root`).
3. **`listen()`**: Переводит сокет в состояние прослушивания (по умолчанию сокет создан, но не готов принимать клиентов). Параметр **backlog** определяет размер очереди клиентов, которые могут «висеть на линии», пока сервер их не принял.
4. **`accept()`**: Блокирующий вызов. Сервер засыпает, пока не придёт клиент. При подключении `accept` возвращает **новый файловый дескриптор** специально для общения с этим клиентом.

> [!IMPORTANT] Важное различие
> На сервере всегда есть **слушающий сокет** (только принимает звонки) и **клиентский сокет** (создаётся для каждого конкретного разговора).

### Чем определяется TCP-соединение?
TCP-соединение однозначно определяется четверкой значений (кортежем):
1. IP-адрес клиента.
2. Порт клиента (назначается ОС случайно при подключении).
3. IP-адрес сервера.
4. Порт сервера.
Благодаря этому сервер может поддерживать множество соединений на одном и том же порту.

### Проблемы однопоточной модели
В простейшем сервере обработка идёт в цикле: принял клиента -> прочитал -> ответил -> закрыл.
* **Проблема:** Если один клиент подключился и молчит, вызов `read` заблокирует весь сервер. Другие клиенты будут стоять в очереди ядра (backlog) и не получат ответа, пока первый не договорит.
* **Решение:** Выносить обработку каждого клиента в отдельный поток (`std::thread)`, а затем использовать пулы потоков и неблокирующий ввод-вывод.

### Проблема C10k

Несмотря на простоту, подход «один поток на одного клиента» имеет серьёзные ограничения, известные как проблема C10k:
- **Затраты на создание потоков:** Каждый поток требует ресурсов ОС и выделения памяти под стек (обычно около 1-8 МБ). 10,000 потоков могут исчерпать оперативную память.
- **Переключение контекста (Context Switch):** Когда потоков слишком много, процессор тратит больше времени на переключение между ними, чем на полезную работу.
- **Масштабируемость:** ОС плохо справляется с управлением десятками тысяч конкурирующих потоков.

#### Инструментарий для новичка
* **`netstat -tan`**: Посмотреть список открытых портов и состояние соединений (`LISTEN`, `ESTABLISHED`).
* **`Netcat (nc)`**: Универсальный швейцарский нож для отправки сырых байтов по TCP/UDP.
* **`telnet`**: Простая утилита для проверки доступности порта.
* **`curl -v`**: Проверка HTTP-серверов с отображением заголовков.
* **`/proc/[pid]/fd`**: Директория в Linux, где можно увидеть все открытые сокеты вашего процесса.

# <h1 id="p2"><center>Примитивы синхронизации</center></h1>

## <h2 id="p2p1"><center>Atomics</center></h2>

==Опр.== **Атомарная операция** - операция, которая выполняется как один неделимый шаг.

==Опр.== **Atomic** - объект, над которым все операции выполняются атомарно.

==Опр.== **Read-Modify-Write (RMW) операции** - комбинированные операции (чтение-изменение-запись), которые выполняются атомарно на аппаратном уровне как единый шаг.

### Свойства Atomic

1. Запрещает компилятору кешировать переменную в регистрах.
2. Гарантирует генерацию процессором специальных инструкций (например, с префиксом `LOCK` в x86: `LOCK CMPXCHG`), которые временно блокируют кэш-линию памяти для других ядер *(протоколы когерентности кэшей вроде MESI)*.
3. Использует RMW-операции.

### Порядок упорядочивания памяти

Atomics не только защищают саму переменную, но и координируют порядок выполнения инструкций вокруг неё через флаги `std::memory_order`:
- `memory_order_relaxed` - гарантирует только атомарность самого значения. Компилятор и процессор могут свободно переставлять окружающие инструкции чтения/записи. Самый быстрый режим.
- `memory_order_release` / `memory_order_acquire` - синхронизирует разные потоки между собой. Все записи в память, сделанные _до_ `release` в Потоке А, гарантированно станут видимы в Потоке Б _после_ того, как он выполнит чтение этой же переменной через `acquire`.
- `memory_order_seq_cst` - режим по умолчанию. Накладывает самый строгий тотальный порядок на все потоки. Безопасен, но наиболее медленный из-за жестких аппаратных ограничений на перестановки.

## <h2 id="p2p2"><center>Mutex</center></h2>

==Опр.== **Критическая секция кода** - часть кода, данные которого должны быть доступны не более чем одному потоку за раз.

==Опр.== **Мьютекс (Mutex - Mutual Exclusion)** - примитив синхронизации, который используются для защиты критических секций, чтобы только один поток имел доступ к данным одновременно.

- **`std::mutex`:** Базовый мьютекс. Методы `lock()` и `unlock()`.
- **`std::recursive_mutex`:** Позволяет одному и тому же потоку захватывать мьютекс несколько раз подряд без возникновения deadlock (взаимной блокировки). Это полезно в рекурсивных функциях.
- **`std::shared_mutex` (C++17):** Реализует логику **Reader-Writer lock**.
    - **Читатели:** Много потоков могут одновременно читать данные (`lock_shared()`), если нет писателя.
    - **Писатели:** Только один поток может писать (`lock()`), блокируя и других писателей, и всех читателей.

### Безопасный вывод: `std::osyncstream` (C++20)

При работе в несколько потоков обычный `std::cout << "..."` может перемешивать сообщения от разных потоков. Оператор `ostream << ...` выполняется атомарно (одним действием), но несколько таких последовательных операций могут конкурировать с другими потоками.

- **Решение:** Использовать `std::osyncstream`. Он накапливает вывод в буфере и записывает его в поток целиком (атомарно) при выходе объекта из области видимости. Пример:
```cpp
std::osyncstream oss{std::cout};
oss << "Hello" << ", " << "world!" << std::endl;
```

### RAII-обёртки для мьютексов

> [!warning] Совет
> Никогда не вызывайте `lock()` и `unlock()` вручную, так как можно забыть освободить мьютекс при ошибке или исключении.

- **`std::lock_guard`:** Максимально простая обёртка: захватывает в конструкторе, освобождает в деструкторе.
- **`std::unique_lock`:** Более гибкая версия. Позволяет вручную разблокировать и снова заблокировать мьютекс, а также передавать владение блокировкой в другие функции.

### Семафоры: `std::counting_semaphore` (C++20)

==Опр.== **Семафор** - это счётчик ресурсов. Его частое применение - ограничение количества одновременных подключений к серверу или управление пулом ресурсов. Обычно он имеет следующие методы:
- **`acquire()`:** Уменьшает счётчик (захватывает ресурс). Если он равен 0, поток засыпает и ждёт.
- **`release()`:** Увеличивает счётчик (освобождает ресурс) и будит ждущие потоки.

**`std::binary_semaphore`:** Семафор со счётчиком от 0 до 1 (аналог мьютекса, но без привязки к конкретному потоку-владельцу).

### Защита от Deadlock при множественном захвате

Если двум потокам нужно захватить два мьютекса (A и B), и первый поток захватывает A, а затем B, тогда как второй поток захватывает B, а затем A - возникает классический Deadlock.

Функция `std::lock(m1, m2, ...)` позволяет захватить два и более мьютекса [[Atomic|атомарно]]. Под капотом она использует внутренний алгоритм, чтобы избежать блокировки. Обычно применяется в связке с отложенными блокировками (C++11) или новым `std::scoped_lock` (C++17):

**Старый стиль (C++11):**
```cpp
std::mutex m1, m2;
std::unique_lock<std::mutex> lock1(m1, std::defer_lock); // Не захватываем сразу
std::unique_lock<std::mutex> lock2(m2, std::defer_lock);
std::lock(lock1, lock2); // Безопасный одновременный захват
```

**Новый стиль (C++17):**
```cpp
std::mutex m1, m2;
// Захватывает любое количество мьютексов без Deadlock
std::scoped_lock lock(m1, m2); 
```

### Важные моменты

1. **Зачем нужен RAII (`lock_guard`)?** 
	Гарантия освобождения ресурсов при исключениях.
2. **Чем семафор отличается от мьютекса?** 
	Семафор - счётчик, у него нет понятия «владельца», его может освободить не тот поток, который его захватил.

## <h2 id="p2p2p1"><center>Spinlock</center></h2>

==Опр.== **Spinlock** - это простейший примитив [[Mutex|mutex]], работающий на принципе активного ожидания (Busy Waiting).

**Принцип работы:** Поток, пытающийся захватить объект, крутится в бесконечном цикле `while`, постоянно проверяя значение разделяемой переменной-флага (например, занято/свободно).

## Реализации Spinlock
### 1. Spinlock на обычном флаге

```c++
class SpinLock {
public:
	void Lock() {
	    while (flag) { /* busy wait */ }
	    flag = true;
	}
	
	void Unlock() {
	    flag = false;
	}

private:
	bool flag = false; 
};
```

Данная реализация имеет критические ошибки и не работает по нескольким причинам:

1) **Оптимизация компилятора** 
	Компилятор видит, что внутри цикла `while (flag) {}` значение `flag` не меняется текущим потоком. Он имеет право закешировать это значение в регистр процессора один раз. В итоге, даже если другой поток изменит `flag` в памяти, ждущий поток об этом никогда не узнает и зависнет в вечном цикле.
    
2) **Data Race**
	Операции проверки `while(flag)` и последующей установки `flag = true` не выполняются неделимо. Два потока могут одновременно увидеть, что `flag == false`, оба выйдут из цикла и оба установят `flag = true`, зайдя в критическую секцию одновременно. Это нарушает инвариант взаимного исключения и приводит к UB.
    
3) **Store Buffer** 
	Каждое ядро процессора имеет сверхбыстрый локальный буфер - **Store Buffer**. Когда поток выполняет операцию записи (`store`), данные физически попадают сначала в этот буфер, а не в L1-кэш или общую оперативную память. Другие ядра увидят изменения только тогда, когда Store Buffer будет сброшен в общую систему кэш-памяти, и сразу несколько потоков одновременно увидят `flag = false` и зайдут в критическую секцию.

### 2. [[Atomic|Атомарный]] Spinlock

Исправляя предыдущие проблемы, мы переходим к использованию atomic и RMW-операций. Метод `.exchange(true)` атомарно записывает `true` и возвращает **старое** значение переменной.
```c++
class SpinLock {
private:
    std::atomic<bool> state_{false};
    
public:
    void Lock() {
        // Атомарно пытаемся установить true. 
        while (state_.exchange(true)) {
            // busy wait
        }
    }
    
    void Unlock() {
        state_.store(false); // Атомарно освобождаем
    }
};
```

#### Как работает?
1. Поток А вызывает `Lock()`. `state_.exchange(true)` видит старое значение `false`, записывает `true`, возвращает `false`. Цикл завершается. Поток А в критической секции.
2. Приходит Поток Б. Вызывает `Lock()`. `exchange(true)` видит текущее значение `true`, записывает `true` и возвращает `true`. Поток Б остается крутиться в цикле.

#### В чём недостаток?
Каждая RMW-инструкция на аппаратном уровне трактуется процессором как операция записи. Согласно протоколам когерентности кэшей *(например, MESI)*, для выполнения записи ядро процессора обязано запросить эксклюзивное владение кэш-линией, содержащей `state_`, и отправить сигнал инвалидации во все остальные ядра. Если 10 потоков будут непрерывно вызывать `exchange(true)`, кэш-линия будет с огромной скоростью «прыгать» от одного ядра к другому по межпроцессорной шине. Этот «шторм» полностью забивает пропускную способность шины памяти, замедляя работу всей системы, включая Поток А, который сейчас находится в критической секции и пытается делать полезную работу.

### 3. TTAS Spinlock

Чтобы убрать шторм инвалидаций кэша, применяется стратегия TTAS (Test and Test-and-Set): «Сначала просто посмотри обычным чтением, а когда освободится - пытайся записать». Обычный метод `.load()` требует кэш-линию только в состоянии _Shared_, что позволяет всем ядрам читать её локально без генерации трафика по шине.
```c++
class TTASSpinLock {
private:
    std::atomic<bool> state_{false};
    
public:
    void Lock() {
        while (true) {
            // Пытаемся захватить замок. Если вернулся false - мы выиграли.
            if (!state_.exchange(true)) {
                return;
            }
            // Если замок занят, крутимся в локальном кэше на обычном чтении
            while (state_.load()) {
                // Spin-loop hint для процессора (опционально)
                #if defined(__x86_64__) || defined(_M_X64)
                _mm_pause(); // Предотвращает конвейерные спекуляции CPU
                #endif
            }
        }
    }
    
    void Unlock() {
        state_.store(false); 
    }
};
```

#### Как работает?
Пока Поток А держит замок, Поток Б крутится во внутреннем цикле `while (state_.load())`. Кэш-линия лежит в L1-кэше ядра Потока Б в состоянии _Shared_. Никакого трафика по шине нет. Как только Поток А делает `Unlock`, кэш-линия инвалидируется **ровно один раз**. Поток Б выходит из внутреннего цикла и делает попытку `exchange(true)`.

### +ы и -ы Spinlock

> [!success] Преимущество
> Если критическая секция выполняется очень быстро *(пару тактов процессора)*, spinlock идеален. Поток не тратит время на переключение в контекст операционной системы. Реакция на освобождение замка - мгновенная.

> [!failure] Недостатки
> 1. **Расход CPU:** Поток сжигает 100% времени одного процессорного ядра на абсолютно холостой цикл.
> 2. **Проблема вытеснения:** Если поток, который уже удерживает spinlock, будет вытеснен планировщиком ОС, все остальные потоки, ждущие этот spinlock, будут безумно крутиться в цикле, тратя впустую миллионы тактов CPU, пока планировщик не вернёт контекст первому потоку.

## <h2 id="p2p2p2"><center>Futex</center></h2>

==Опр.== **Futex (Fast Userspace Mutex)** - это базовый механизм синхронизации в операционных системах, который служит фундаментом для построения привычных примитивов: `std::mutex`, `std::condition_variable`, семафоров и т.д.

- Простыми словами, futex - это **гибридный механизм**, который объединяет скорость работы в userspace и мощь ядра ОС для укладывания потоков в спячку.

### Какую проблему решает futex?

До появления futex у разработчиков было два принципиальных (и крайних) пути синхронизации потоков, у каждого из которых были огромные минусы:

1. **Блокировки через ядро:**
    При любой попытке взять заблокированный ресурс поток шёл в ядро ОС через системный вызов, и ядро решало, усыпить поток или дать ему управление.
    - **Минус:** Системный вызов - это очень дорогая операция *(сотни наносекунд, переключение контекста, сброс кэшей процессора)*. Вызывать его, когда объект свободен и конкуренции потоков прямо сейчас нет - непозволительная роскошь.

2. **Spinlocks:**
    Поток крутится в бесконечном цикле (`while`), проверяя атомарную переменную, пока она не станет свободной. Всё происходит в userspace, без захода в ядро.
    - **Минус:** Если ресурс занят надолго, ждущий поток полностью загружает ядро процессора («выжигает» CPU), выполняя бесполезную работу.


> [!tip] Идея futex 
> А что если мы будем использовать atomic в пространстве пользователя, когда конкуренции нет, и будем обращаться к ядру ОС _только тогда, когда реально нужно уснуть или кого-то разбудить_?

### Как работает futex?

Конструктивно ftex состоит из двух частей:

1. **Обычное 32-битное целое число (`int32_t`)** в памяти программы *(в userspace)*.
2. **Очередь ожидания внутри ядра ОС**, привязанная к физическому адресу этого числа.

А работа с futex делится на два сценария:
#### 1. Быстрый путь - Конкуренции нет

Поток хочет захватить mutex. Он выполняет быструю атомарную инструкцию процессора над этим 32-битным числом прямо в userspace. Если число показывало «свободно», поток атомарно меняет его на «занято» и идёт дальше. **Никаких системных вызовов к ядру не происходит.**

#### 2. Медленный путь - Конкуренция есть

Если при выполнении атомарной операции поток видит, что mutex уже занят другим потоком, он не хочет бесконечно крутиться в цикле. Вот здесь приложение вызывает системный вызов и просит ядро: _«Усыпи меня, пока адрес этого числа не освободится»_. Ядро убирает поток из планировщика, и тот больше не тратит ресурсы процессора.

### Низкоуровневые вызовы ядра: `WAIT` и `WAKE`

Интерфейс ядра Linux предоставляет системный вызов `sys_futex`, у которого есть две базовые операции:

1. **`FUTEX_WAIT(volatile uint32_t* uaddr, uint32_t val)`**
    Поток говорит ядру: _«Я хочу уснуть на очереди у этого `uaddr`. Но перед тем как усыпить меня, проверь, что там всё ещё лежит значение `val`»_.
    - **Зачем нужна проверка?** Это защита от состояния гонки. Пока поток готовился сделать системный вызов, другой поток мог уже отпустить мьютекс и поменять значение. Если значение в памяти уже не равно `val`, `FUTEX_WAIT` сразу возвращает ошибку, не укладывая поток спать зря.

2. **`FUTEX_WAKE(volatile uint32_t* uaddr, int num_waiters)`**
    Когда поток освобождает ресурс, он меняет значение переменной в userspace и делает вызов `FUTEX_WAKE`, приказывая ядру: _«Разбуди максимум `num_waiters` потоков, которые спят на этом `uaddr`»_. Ядро находит нужную очередь и возвращает потоки в строй.

### Ложные пробуждения

==Опр.== **Ложное пробуждение** - пробуждение потока, который вызывал `FUTEX_WAIT` без вызова `FUTEX_WAKE`. Это может происходить из-за сигналов ОС или внутренних особенностей ядра.

==Опр.== **Потерянное пробуждение (Lost Wake-up)** - баг, при котором не обрабатывается ложное пробуждение и поток, который проснулся потерян.

- **Ложные пробуждения (Spurious Wakeups):** Поток, вызвавший `FUTEX_WAIT`, может проснуться не потому, что его разбудил `FUTEX_WAKE`, а из-за сигналов ОС или внутренних особенностей ядра. Именно поэтому проверку futex и захват блокировки всегда оборачивают в цикл (`while`), повторно проверяя условие через atomics после просыпания.
- **Futex - это не замок сам по себе**, это механизм контролируемого засыпания потока по условию равенства значения в памяти.

## <h2 id="p2p2p3"><center>Реализация mutex на futex</center></h2>

[[Spinlock]] сжигают 100% CPU, если критическая секция долгая. Для долгих задач нужен mutex, который взаимодействует с операционной системой через системный вызов [[Futex]], усыпляя потоки в ядре ОС, когда критическая секция занята.

## Реализации
### 1. Двухпозиционный Futex-mutex

Если строить мьютекс на простом булевском флаге (0 - свободен, 1 - занят), то при каждом вызове `unlock()` мы будем обязаны вызывать системный вызов `futex_wake`, чтобы проверить, нет ли спящих. 

```c++
class NaiveFutexMutex {
private:
    std::atomic<int> state_{0}; // 0 - unlocked, 1 - locked
public:
    void Lock() {
        // Крутимся, пока не сможем атомарно перевести из 0 в 1
        while (state_.exchange(1) == 1) {
            // Уходим спать в ядро ОС. 
            futex_wait(&state_, 1); 
        }
    }
    void Unlock() {
        state_.store(0);
        // Сискол wake вызывается ВСЕГДА!
        futex_wake(&state_, 1); 
    }
};
```

- **Как futex защищает от Lost Wake-up:**
    _Сценарий гонки:_ Поток Б проверил `while`, замок занят. Поток Б собирается вызвать `futex_wait`. Прямо перед системным вызовом контекст переключается. Поток А делает `Unlock()`: сбрасывает `state_` в `0` и делает `futex_wake`. Очередь пуста, сигнал улетает в никуда. Контекст возвращается к Потоку Б, и он вызывает `futex_wait(&state_, 1)`. Системный вызов `futex_wait` внутри ядра атомарно перепроверяет: `if (*addr == val)`. Ядро видит, что в памяти по адресу `&state_` уже лежит 0, а мы передали ожидаемое значение 1. Проверка проваливается, ядро мгновенно возвращает управление Потоку Б с ошибкой `EWOULDBLOCK`, не укладывая его спать. Поток Б делает следующий круг цикла и забирает замок.
    
- **Критический недостаток этой реализации:**
    В методе `Unlock()` системный вызов `futex_wake` происходит **при каждом освобождении мьютекса**, даже если за замком никто не стоит. Теряется весь смысл концепции "Fast Userspace", где в отсутствие конкуренции всё должно работать без syscall.


### 2. Mutex на 3 состояниях

Чтобы полностью исключить системные вызовы на `Unlock()`, если конкуренции нет, состояние мьютекса расширяют до трёх значений:
- **`0` (Unlocked):** Мьютекс полностью свободен.
- **`1` (Locked, no waiters):** Мьютекс занят одним потоком, но за ним в очереди больше никто не стоит.
- **`2` (Locked, with waiters):** Мьютекс занят, и есть другие потоки, которые заблокировались и ушли спать в ядро.

#### Как это оптимизирует работу?

- **При захвате (Lock):** Поток пытается атомарно заменить `0` на `1`. Если получилось - мы захватили блокировку без syscall. Если там уже была `1` или `2`, поток переводит состояние в `2` (указывая, что теперь точно есть ждущие) и вызывает `FUTEX_WAIT`.

- **При освобождении (Unlock):** Поток атомарно уменьшает значение или делает CAS.
    - Если состояние было `1` (никто не ждал), поток просто сбрасывает его в `0`. **Системный вызов `FUTEX_WAKE` не вызывается!** Мы сэкономили кучу времени.
    - Если состояние было `2` (кто-то спит в ядре), поток сбрасывает его в `0` и **обязательно вызывает `FUTEX_WAKE`**, чтобы разбудить спящего товарища.

#### Код реализации
```c++
class FutexMutex3States {
private:
    std::atomic<int> state_{0};

public:
    void Lock() {
        int expected = 0;
        // Если mutex полностью свободен, атомарно переводим его в состояние 1
        if (state_.compare_exchange_strong(expected, 1, std::memory_order_acquire)) {
            return;
        }

        // Встаём в очередь, т.к. mutex уже занят
        if (expected != 2) {
            expected = state_.exchange(2, std::memory_order_acquire);
        }

        // Входим в цикл ожидания
        while (expected != 0) {
            // Уходим спать в ядро ОС
            futex_wait(&state_, 2);
            
            // После пробуждения пытаемся снова захватить мьютекс
            expected = state_.exchange(2, std::memory_order_acquire);
        }
    }

    void Unlock() {
        // Атомарно освобождаем замок, записывая 0, и смотрим на предыдущее состояние.
        // Если оно было 1, значит никто не пришёл и не встал в очередь.
        // Мы просто обнулили переменную и сэкономили кучу времени на системном вызове!
        // Если же оно было 2, значит, в ядре гарантированно есть спящие потоки.
        if (state_.exchange(0, std::memory_order_release) == 2) {
            // Вызываем пробуждение для 1 потока
            futex_wake(&state_, 1);
        }
    }
};
```

## Низкоуровневые сценарии конкуренции
### Сценарий 1: Быстрый путь
1. **Захват (`Lock`):** Поток А приходит к свободному мьютексу (переменная равна `0`). Он выполняет атомарную RMW-операцию, пытаясь установить значение `1`. Операция возвращает старое значение `0`. Поток А понимает, что замок успешно взят, и мгновенно заходит в критическую секцию. 
2. **Освобождение (`Unlock`):** Поток А закончил работу. Он атомарно проверяет состояние. Поток А видит, что состояние мьютекса по-прежнему равно `1` (никто не пришел, пока он работал). Он просто сбрасывает состояние в `0`. Мы получили скорость spinlock в отсутствие конкуренции!

### Сценарий 2: Медленный путь (Появление конкурента)
1. Поток А удерживает мьютекс (состояние равно `1`).
2. Приходит Поток Б, выполняет атомарный `exchange(1)`. Назад возвращается `1`. Поток Б понимает: _«Замок занят Потоком А»_.
3. Поток Б осознает, что ему нужно идти спать в ядро. Но перед этим он обязан гарантировать, что поток А узнает о его присутствии. Поток Б с помощью атомарной операции принудительно переводит состояние мьютекса в `2`.
4. Поток Б вызывает `futex_wait(addr, 2)`. Ядро Linux проверяет: _«Правда ли там сейчас лежит двойка?»_. Да, поток Б только что записал её. Ядро усыпляет Поток Б и забирает у него управление процессором.
5. Поток А выходит из критической секции. Делает `state_.exchange(0)`. Ему возвращается старое значение `2`. Поток А понимает: _«За мной заняли очередь в ядре»_. Он обязан вызвать `futex_wake(&state_, 1)` для пробуждения Потока Б.

### Сценарий 3: Феномен накатывания
1. Поток А вызвал `unlock`, сбросил состояние в `0` и сделал `futex_wake`. Поток Б (который спал) начинает процедуру пробуждения в недрах ОС.
2. Пробуждение из ядра занимает время *(планировщику нужно выделить квант времени, переключить контекст)*. Поток Б ещё физически не выполнил ни одной строчки кода в Userspace.
3. В этот самый микросекундный момент на процессорном ядре прорывается абсолютно новый, активный **Поток В**. Он вызывает метод `lock()`, делает атомарный `exchange` и видит, что состояние мьютекса равно `0`.
4. Поток В мгновенно захватывает мьютекс, меняет состояние на `1` (или `2`) и уходит выполнять критическую секцию.
5. **Что происходит со спящим Потоком Б, когда он окончательно «открыл глаза»:** Поток Б выходит из системного вызова `futex_wait`. Он заходит на следующую итерацию своего внутреннего цикла ожидания `while` и пытается атомарно забрать замок. Но он видит, что значение уже НЕ 0.
6. Поток Б понимает, что замок опять занят, атомарно выставляет состояние мьютекса обратно в `2` и послушно уходит спать назад в `futex_wait`.

_Такое поведение называется **Barging (накатывание / нечестный мьютекс)**. Мьютекс получается нечестным (Поток Б ждал долго, а Поток В проскочил без очереди), но это колоссально увеличивает общую пропускную способность системы. Процессор не простаивает ни одного такта, пока Поток Б медленно просыпается из ядра._

## <h2 id="p2p3"><center>Condvar</center></h2>

Представьте сценарий, когда одному потоку нужно дождаться выполнения некоторого условия, которое должен изменить другой поток (например, дождаться, пока в очереди появится задача для обработки). Как такое можно реализовать?

> [!fail] Неэффективный подход (Busy Waiting)
> Поток крутится в бесконечном цикле проверки флага: `while (!ready) {}`. При этом ядро процессора загружается на 100%, выполняя абсолютно бесполезную работу по постоянному опросу переменной.

> [!done] Правильно:
> Использовать условную переменную (condvar).

==Опр.== **Условная переменная (Condition variable - condvar)** - это механизм синхронизации, позволяющий потоку заблокироваться (заснуть, полностью освободив процессор) до тех пор, пока другой поток не изменит состояние данных и не оповестит об этом.

### Как это устроено в C++?

Для корректной работы условной переменной всегда требуются три компонента:
1. Сама условная переменная: `std::condition_variable cv;`
2. Мьютекс, защищающий разделяемое состояние: `std::mutex mtx;`
3. Переменная состояния (предикат), отражающая условие (например, `bool ready = false;` или размер очереди `tasks.empty()`).

#### Ключевые методы:

- `wait((std::unique_lock<std::mutex>& lock))`: усыпляет текущий поток. При вызове `wait` поток _атомарно отпускает мьютекс_ и засыпает. Это нужно, чтобы другие потоки могли захватить мьютекс и изменить условие. Как только поток пробуждается, он _автоматически заново захватывает мьютекс_ перед выходом из функции `wait`.
- `notify_one()`: будит ровно один из потоков, ожидающих на данной условной переменной.
- `notify_all()`: будит абсолютно все потоки, которые в данный момент заблокированы на этой условной переменной.

### Ложные пробуждения (Spurious Wakeups)

==Опр.== **Ложное пробуждение** - это ситуация, когда поток выходит из состояния ожидания сам по себе, даже если никто не вызывал `notify_one()` или `notify_all()`, либо если условие на самом деле еще не выполнено.

#### Почему это происходит?

Это обусловлено архитектурой ядер современных операционных систем *(включая Linux и Windows)*. Гарантия полного отсутствия ложных пробуждений на уровне ОС потребовала бы колоссального усложнения кода ядра и сильно замедлила бы работу примитивов синхронизации. Стандарты API *(POSIX threads и C++)* осознанно допускают ложные пробуждения, перекладывая обязанность повторной проверки условия на программиста.

#### Пример улучшения кода (Борьба с ложным пробуждением)

> [!bug] Плохой код:
```c++
std::unique_lock<std::mutex> lock(mtx);
if (!ready) {} // Использование условия if вместо цикла while!
// ОШИБКА: Если произойдет ложное пробуждение, код пойдет дальше, UB.
```

> [!done] Хороший код:
```c++
std::unique_lock<std::mutex> lock(mtx);
while (!ready) { // Всегда проверяем условие в цикле while!
    cv.wait(lock); // При ложном пробуждении поток вернется к проверке ready
}
// Гарантировано: мы пройдём сюда только если ready == true.
```

> [!tip] Синтаксический сахар
> В C++ есть переопределённый метод `wait`, автоматически оборачивающий ожидание в цикл `while`:
```c++
cv.wait(lock, [] { return ready; }); // Передача лямбда-предиката
```

### Реализация [[Mutex#Семафоры `std counting_semaphore` (C++20)|семафора]] через Convar

```c++
class CorrectSemaphore {
private:
    std::mutex mtx;
    std::condition_variable cv;
    int count;
public:
    explicit CorrectSemaphore(int init_count) : count(init_count) {}

    void acquire() {
        std::unique_lock<std::mutex> lock(mtx);
        // Защищаемся от ложных пробуждений: спим до тех пор, пока ресурс равен 0
        while (count == 0) {
            cv.wait(lock);
        }
        count--;         // Забираем один ресурс
    }

    void release() {
        std::unique_lock<std::mutex> lock(mtx);
        count++;         // Освобождаем ресурс
        cv.notify_one(); // Оповещаем один ждущий поток
    }
};
```

### Важные моменты

1. Почему в метод `cv.wait()` мы обязаны передавать именно `std::unique_lock`, а не более лёгкий `std::lock_guard`?
    - `std::lock_guard` не поддерживает ручной вызов методов блокировки и разблокировки. `std::condition_variable::wait()` устроен так, что ему необходимо самостоятельно отпускать мьютекс при засыпании и захватывать его обратно при пробуждении. На это способен только `std::unique_lock`.

2. Когда использовать `notify_one()`, а когда `notify_all()`?
    - `notify_one()` используют, когда все ждущие потоки выполняют эквивалентные задачи и только один из них сможет обработать изменение состояния. `notify_all()` используется, когда потоки ждут разных условий или когда логика требует пробуждения всех *(например, при завершении работы пула или в барьерах синхронизации)*.

## <h2 id="p2p3p1"><center>Реализация condvar на futex</center></h2>

## 1. Наивная реализация

Если попытаться написать [[Condvar|condvar]] тривиально, используя системный вызов [[Futex|futex]], мы получим классическое состояние гонки. Представим наивный алгоритм `Wait`:

1. Поток видит, что условие не выполнено.
2. Он вызывает `lock.unlock()`, освобождая mutex.
3. Он вызывает `futex_wait(&cv_word, 0)`.

#### Сценарий гонки:
1. Поток А выполняет шаг 2 (отпускает мьютекс), но ещё не успевает войти в ядро ОС для вызова `futex_wait`.
2. В этот микросекундный момент Поток Б захватывает мьютекс, меняет предикат и вызывает `NotifyOne`, который транслируется в `futex_wake(&cv_word, 1)`.
3. Так как Поток А ещё не спит в ядре, очередь ожидания futex пуста. Сигнал пробуждения уходит «в никуда».
4. Поток А возвращает себе управление, заходит в ядро и вызывает `futex_wait`. Он засыпает **навсегда**, хотя данные для него уже готовы.

## 2. Механизм счётчика поколений

Чтобы исключить утерянное пробуждение, CV обязана отслеживать «номера поколений» *(sequence numbers)* нотификаций. Для этого заводится переменная `std::atomic<uint32_t> generation_{0}`.

```c++
class Condvar {
private:
    std::atomic<uint32_t> generation_{0}; // Глобальный счётчик поколений

public:
    template <typename Mutex>
    void Wait(Mutex& lock) {  // метод вызывается под lock.Lock
        uint32_t current_gen = generation_.load(std::memory_order_relaxed);

        lock.Unlock();
        // Если перед syscall сменилось поколение, метод вернёт ошибку EWOULDBLOCK
        futex_wait(&generation_, current_gen);
        lock.Lock();
    }

    void NotifyOne() {        // метод вызывается под lock.Lock
        generation_.fetch_add(1, std::memory_order_seq_cst);
        futex_wake(&generation_, 1);
    }

    void NotifyAll() {
        generation_.fetch_add(1, std::memory_order_seq_cst);
        futex_wake(&generation_, INT_MAX);
    }
};
```

## 3. Оптимизация производительности

Если поток вызывает `cv.notify_one()` внутри критической секции, возникает **проблема Mesa-style scheduling**:
- Проснувшийся в ядре ждущий поток мгновенно пытается захватить mutex, но mutex всё ещё держится нотифицирующим потоком!
- Ждущий поток бьётся о закрытый замок mutex и тут же снова блокируется внутри ОС (уходит спать в очередь мьютекса). Происходят два лишних и дорогостоящих переключения контекста.

**Решение (`FUTEX_CMP_REQUEUE`):**
Современные condvar в Linux используют системный вызов `futex` с флагом `FUTEX_CMP_REQUEUE`. Вместо того чтобы физически будить потоки из очереди condvar, ядро ОС **напрямую перевешивает** спящие потоки из очереди ожидания условной переменной в очередь ожидания mutex внутри самого ядра. Потоки просыпаются только тогда, когда mutex реально освобождается.

## <h2 id="p3p1"><center>Thread pool</center></h2>

==Опр.== **Thread Pool (Пул потоков)** - объект, создающий фиксированное количество рабочих потоков, которые живут всё время работы приложения и повторно используются для выполнения поступающих задач.

> [!tip] Оптимизация
> Thread pool решает [[Жизненный цикл TCP-сервера#Проблема C10k|проблему C10k]], ускоряя обработку данных.

### Полная реализация Thread Pool через [[Condvar|условную переменную]]

```c++
#include <vector>
#include <queue>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <functional>

class ThreadPool {
private:
    std::vector<std::thread> workers;          // Пул созданных потоков
    std::queue<std::functional<void()>> tasks; // Очередь задач
    
    std::mutex queue_mutex;                    // Мьютекс для защиты очереди задач
    std::condition_variable cv;                // Сигнал для засыпания/пробуждения работников
    bool stop = false;                         // Флаг остановки пула

public:
    ThreadPool(size_t threads) {
        for (size_t i = 0; i < threads; ++i) {
            workers.emplace_back([this] {
                while (true) {
                    std::functional<void()> task;
                    {
                        std::unique_lock<std::mutex> lock(this->queue_mutex);
                        // Поток спит, пока не появится задача или сигнал стоп
                        this->cv.wait(lock, [this] { 
                            return this->stop || !this->tasks.empty(); 
                        });
                        
                        // Если пул закрывается и задач не осталось - завершаем поток
                        if (this->stop && this->tasks.empty()) {
                            return;
                        }
                        
                        // Забираем задачу из очереди
                        task = std::move(this->tasks.front());
                        this->tasks.pop();
                    } 
                    // Выполняем задачу ВНЕ лока мьютекса!
                    task(); 
                }
            });
        }
    }

    // Добавление новой задачи в пул
    void enqueue(std::functional<void()> task) {
        {
            std::unique_lock<std::mutex> lock(queue_mutex);
            tasks.push(task);
        }
        cv.notify_one(); // Будим один свободный поток
    }

    ~ThreadPool() {
        {
            std::unique_lock<std::mutex> lock(queue_mutex);
            stop = true;
        }
        cv.notify_all(); // Пробуждаем все потоки для корректного завершения
        for (std::thread &worker : workers) {
            if (worker.joinable()) {
                worker.join();
            }
        }
    }
};
```

### Фундаментальная уязвимость наивного Thread Pool

- Thread Pool всё ещё уязвим при использовании **блокирующего ввода-вывода**.

#### Сценарий «Атаки N клиентов»:

Представьте, что в запущенном TCP-сервере выделено в Thread Pool ровно N потоков. Каждое входящее соединение передаёт функцию `handle_client` в пул задач. Внутри этой функции происходит стандартное блокирующее чтение из сети: `read(client_socket, buf, size);`

Если злоумышленник *(или N медленных клиентов)* откроет одновременно N соединений и **не отправит ни одного байта данных**, произойдёт катастрофа:

1. Все N потоков пула возьмут эти N задач.
2. Каждый поток зайдёт в системный вызов `read()`.
3. Поскольку данных нет, а вызов блокирующий, операционная система усыпит эти потоки до момента появления данных.
4. **Результат:** Пул потоков полностью парализован *(все работники заняты вечным ожиданием)*. Когда к серверу подключится (N+1)-й легитимный клиент, его задача встанет в очередь, но её абсолютно некому будет обработать. **Сервер уходит в отказ в обслуживании**.

> [!warning] Вывод
> Thread Pool эффективен для вычислительных задач (CPU-bound) или неблокирующего I/O. Использование блокирующих сетевых вызовов внутри фиксированного пула потоков делает систему уязвимой к deadlock ресурсов.

### Важные моменты

1. Почему в реализации Thread Pool вызов `task()` происходит строго после выхода из блока мьютекса?
    - Если запустить `task()` под заблокированным мьютексом очереди, то на всё время выполнения этой задачи очередь будет заблокирована. Другие потоки пула не смогут забрать свои задачи из очереди, и параллельное выполнение превратится в строго последовательное (однопоточное).

## <h2 id="p3p1p1"><center>Busy Loop Polling Server</center></h2>

Ранее мы выяснили, что создание потока на каждого клиента приводит к нехватке ресурсов ОС (проблема C10k), и разработали [[Thread Pool]]. Однако, если мы используем базовый блокирующий ввод-вывод, потоки из пула быстро "повиснут" на системном вызове `read()`, ожидая данных от медленных клиентов. Пул истощится, и сервер остановится. Для решения этой проблемы сокеты переводятся в неблокирующий режим с помощью `fcntl`:
```cpp
int flags = fcntl(client_fd, F_GETFL, 0);
fcntl(client_fd, F_SETFL, flags | O_NONBLOCK);
```
Теперь поведение `read` и `write` меняется:
- Если данные есть - функция забирает их и возвращает количество байт.
- Если данных нет - функция не усыпляет поток, а **моментально возвращает `-1`**, при этом переменная `errno` устанавливается в значение `EAGAIN` или `EWOULDBLOCK` ("ресурс временно недоступен, попробуйте снова").

Сервер может использовать один поток, который хранит список всех подключенных клиентов (в `std::vector<int>`\) и в бесконечном цикле опрашивает их.

```cpp
// ... accept non-blocking сокетов и добавление в vector clients ...
while (server_is_running) {
    for (auto it = clients.begin(); it != clients.end(); ) {
        char buffer[1024];
        ssize_t bytes = read(*it, buffer, sizeof(buffer));

        if (bytes > 0) {
            // Данные получены, обрабатываем!
            work();
            ++it;
        } else if (bytes == -1 && (errno == EAGAIN || errno == EWOULDBLOCK)) {
            // Данных пока нет, идем дальше по списку
            ++it;
        } else {
            // Ошибка или EOF
            close(*it);
            it = clients.erase(it);
        }
    }
}
```

#### Проблемы такой реализации:
1. **Busy Waiting:** Поток ни на секунду не засыпает. Даже если сервер простаивает и никто ничего не пишет, ядро процессора будет загружено на 100%, выполняя бессмысленные `read`, возвращающие `EAGAIN`.
2. **Плохая масштабируемость ($O(N)$):** Если у нас подключено 10 000 клиентов, а данные пришли только от одного, потоку нужно сделать 10 000 дорогих системных вызовов, чтобы найти этого одного клиента.

## <h2 id="p3p1p2"><center>Nonlocking thread pool</center></h2>

Чтобы избавиться от [[Busy Loop Polling Server|Busy Loop]], нам нужен механизм, позволяющий сказать ядру ОС: _"Усыпи мой поток и разбуди, только когда хотя бы на одном из этих N сокетов появятся данные"_. Для этого служат мультиплексоры I/O.

## 1. Системный вызов `select`

Исторически первый инструмент. Ядру передаются битовые маски сокетов (`fd_set`) для отслеживания.

**Проблемы:** 
* Максимальный файловый дескриптор жёстко ограничен константой `FD_SETSIZE` (обычно 1024). Для решения проблемы C10k не подходит в принципе.
- На каждый вызов приходится заново копировать битовые маски из user-space в ядро и обратно, так как ядро их затирает.
- После возврата `select` потоку нужно линейно перебрать все 1024 бита ($O(N)$), чтобы понять, какой именно сокет готов.

## 2. Системный вызов `poll`

Исправляет ограничение `select`. Принимает массив структур `pollfd`.
- **Плюсы:** Нет ограничения в 1024 сокета.
- **Проблемы:** Огромные накладные расходы на копирование. Если клиентов 10 000, каждый вызов `poll` требует скопировать весь этот массив в ядро. Найти сработавший сокет всё ещё требует итерации по всему массиву ($O(N)$).

## 3. Системный вызов `epoll`

Современный стандарт для высокопроизводительных серверов на Linux. В отличие от предыдущих вызовов, `epoll` сохраняет состояние внутри ядра.

- `epoll_create()`: Создаёт внутри ядра структуру (на базе красно-чёрного дерева для быстрого поиска и очереди готовых событий).
- `epoll_ctl()`: Добавляет, изменяет или удаляет сокеты для мониторинга ($O(\log N)$). Вызывается только один раз при подключении/отключении клиента. Не нужно копировать весь список каждый раз!
- `epoll_wait()`: Блокирует поток до появления активности. **Самое важное:** возвращает массив _только тех сокетов, на которых реально произошли события_ ($O(K)$, где $K$ - число активных сокетов). Никакого $O(N)$ поиска!

#### Реализация многопоточного сервера через `epoll` (Паттерн Reactor):

Один выделенный поток (Reactor) крутит цикл с `epoll_wait()`. Как только `epoll_wait` возвращает список "готовых" дескрипторов, Reactor не обрабатывает их сам, а формирует задачи (например, `ReadTask(fd)`) и кладет их в очередь Thread Pool. Рабочие потоки из пула достают задачу, делают один успешный (неблокирующий) `read`, так как epoll уже гарантировал наличие данных, и отправляют ответ.

```cpp
constexpr int MAX_EVENTS = 64;

int main() {
    int server_sock = socket(AF_INET, SOCK_STREAM, 0);
    // ... (bind, listen) ...

    int epfd = epoll_create1(0); // Создаём epoll

    epoll_event ev{};
    ev.events = EPOLLIN; // Нас интересует чтение (входящие соединения)
    ev.data.fd = server_sock;
    epoll_ctl(epfd, EPOLL_CTL_ADD, server_sock, &ev); // Добавляем сервер в epoll

    std::vector<epoll_event> events(MAX_EVENTS);

    while (true) {
        // Блокируемся до появления событий. Возвращает количество готовых FD
        int ready = epoll_wait(epfd, events.data(), MAX_EVENTS, -1);
        if (ready < 0) break;

        // Итерируемся ТОЛЬКО по готовым событиям (O(K))
        for (int i = 0; i < ready; ++i) {
            int fd = events[i].data.fd;

            if (fd == server_sock) {
                // Новое подключение
                int client_fd = accept(server_sock, nullptr, nullptr);
                
                epoll_event client_ev{};
                client_ev.events = EPOLLIN; // В многопоточном пуле добавили бы | EPOLLET
                client_ev.data.fd = client_fd;
                epoll_ctl(epfd, EPOLL_CTL_ADD, client_fd, &client_ev);
            } else {
                // Готово для чтения
                char buffer[1024];
                ssize_t bytes = read(fd, buffer, sizeof(buffer));
                if (bytes > 0) {
                    // Эхо
                    write(fd, buffer, bytes);
                } else {
                    // Клиент отключился. Закрытие FD автоматически удаляет его из epoll
                    epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
                    close(fd);
                }
            }
        }
    }
    close(server_sock);
    close(epfd);
}
```

## 4. Механизм `io_uring`

**Проблема `epoll`:** Это **синхронный** механизм. `epoll` только говорит: _"Данные есть, иди читай"_. Потоку всё равно приходится делать syscall `read`, переключать контекст в режим ядра, чтобы ядро скопировало данные из буфера сетевой карты в буфер приложения.

`io_uring` - это **истинно асинхронный** ввод-вывод. Здесь мы говорим ядру: _"Вот буфер в user-space. Самостоятельно считай данные из этого сокета сюда и сообщи мне, когда закончишь"_.

**Фундаментальное отличие `io_uring` от `epoll`:**
При инициализации `io_uring` ядро через `mmap` отображает участок оперативной памяти напрямую в адресное пространство процесса *(Shared Memory)*. В этой памяти создаётся два кольцевых буфера:
1. **SQ (Submission Queue - Очередь отправки):** Сюда приложение кладет задания для ядра (SQE - Submission Queue Entries), например "сделай read на этом сокете в этот буфер".
2. **CQ (Completion Queue - Очередь завершения):** Сюда ядро кладет результаты (CQE), когда I/O операция физически завершена.

Формат SQE:
```c
struct request_info {
    int type; // ACCEPT, READ, WRITE...
    int fd;
    char buffer[];
    int buffer_size;
};
```

- Синхронизация указателей головы и хвоста буферов-очередей происходит через барьеры памяти atomic (Acquire/Release семантика) без системных блокировок.

### +ы & -ы `io_uring`


> [!success] Преимущества
> 1. **Отсутствие системных вызовов**. Благодаря shared memory можно отправлять сотни запросов и забирать сотни результатов _вообще без переключений контекста_, что даёт колоссальную пропускную способность.
> 2. **Унифицированность**. `io_uring` идеально и единообразно асинхронно работает как с сетью, так и с диском.
> 3. **Работа с батчами**. Можно положить батч запросов `read` и `write` в SQ и сделать всего один вызов `io_uring_enter`.

> [!failure] Недостатки
> 4. Требует *(относительно)* новых версий ядра ОС.
> 5. Сложнее в настройке и использовании (много ручной работы с барьерами памяти).
> 6. Имеет проблемы с безопасностью *(в контейнерах Kubernetes/Docker доступ к io_uring часто по умолчанию заблокирован из-за прямого доступа к памяти ядра)*.

### Реализация сервера с `io_uring`

```c++
void run_server(int server_fd) {
    struct io_uring ring;
    io_uring_queue_init(256, &ring, 0); // Инициализация очередей

    // 1. Просим ядро сделать асинхронный accept
    struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
    struct request_info *req = malloc(sizeof(*req));
    req->type = 1; // ACCEPT
    req->fd = server_fd;

    io_uring_prep_accept(sqe, server_fd, NULL, NULL, 0);
    io_uring_sqe_set_data(sqe, req); // Привязываем наш контекст
    io_uring_submit(&ring);          // Уведомляем ядро

    struct io_uring_cqe *cqe;
    while (1) {
        // Ждем результата от ядра
        io_uring_wait_cqe(&ring, &cqe);
        struct request_info *completed_req = io_uring_cqe_get_data(cqe);
        int res = cqe->res; // Для accept() это client_fd

        if (completed_req->type == 1 && res > 0) { // Это был accept
            int client_fd = res;
            
            // 2. Сразу формируем новую задачу READ для этого клиента
            struct io_uring_sqe *read_sqe = io_uring_get_sqe(&ring);
            struct request_info *read_req = malloc(sizeof(*read_req));
            read_req->type = 2; // READ
            read_req->fd = client_fd;

            // Говорим ядру: прочитай данные ПРЯМО В read_req->buffer!
            io_uring_prep_read(read_sqe, client_fd, read_req->buffer, 1024, 0);
            io_uring_sqe_set_data(read_sqe, read_req);
            
            // И перезаряжаем ACCEPT для новых клиентов...
            // ...
            io_uring_submit(&ring);
        } else if (completed_req->type == 2 && res > 0) {
            // READ завершился, данные УЖЕ в completed_req->buffer
            // Формируем задачу WRITE...
        }

		// Сдвигаем Head в CQ (отмечаем как обработанное)
        io_uring_cqe_seen(&ring, cqe);
        free(completed_req);
    }
}
```

## <h2 id="p3p2"><center>Future&Promise</center></h2>

В традиционном синхронном программировании вызов функции блокирует поток до тех пор, пока результат не будет вычислен. В конкурентном программировании для разделения во времени момента **запуска** задачи и момента **получения её результата** используется пара примитивов: `Future` и `Promise`.

==Опр.== **Future** - это объект-клиент, предоставляющий интерфейс для доступа к результату асинхронной операции, который, возможно, ещё не вычислен.
Основные методы концептуальной `Future`:
- `IsReady()` - возвращает `true/false`, готова ли задача *(вычисляется мгновенно без блокировки)*.
- `Wait()` - синхронно блокирует текущий поток до тех пор, пока результат не станет доступен.
- `GetResult()` *(или `.get()`)* - возвращает вычисленное значение *(если оно не готово, выполняет `Wait()` пока оно не будет готово)*.

==Опр.== **Promise** - это объект-сервер, представляющий собой принимающий конец канала связи. Поток, выполняющий асинхронную задачу, использует `Promise` для записи результата или ошибки.
Основные методы концептуального `Promise`:
- `SetValue(value)` - сохраняет вычисленный результат и сигнализирует ассоциированной `Future` о готовности.
- `SetException(exception_ptr)` - сохраняет информацию об ошибке/исключении, если в процессе выполнения задачи что-то пошло не так.

> [!tip] Применение Future для асинхронного выполнения задач
> Когда поток вызывает функцию, она не блокирует его, а мгновенно создаёт `Promise`, извлекает из него связанную `Future` через метод `getFuture()` и возвращает её вызывающему коду. Сама тяжёлая задача уходит на выполнение в фоновый поток. Вызывающий поток может параллельно выполнять другую полезную работу, а затем, когда результат действительно понадобится, вызвать метод получения значения.

## Реализация классов `std::future` и `std::promise`

Классы `std::promise` и `std::future`, появившиеся в стандарте C++11, внутри себя не содержат данных напрямую. Они являются легковесными обёртками-указателями на скрытый разделяемый объект, распределённый в динамической памяти - **Shared State**.

Shared State инкапсулирует в себе:
```c++
template<typename T>
struct SharedState {
	std::variant<std::optional<T>> value;
	std::variant<std::exception_ptr> exception;
	std::atomic<bool> is_ready;
	std::mutex mtx;
	std::condition_variable cv;
}
```

#### Алгоритм взаимодействия:

- Когда рабочий поток вызывает `promise.set_value()`, он захватывает mutex внутри Shared State, записывает данные, выставляет флаг `is_ready = true` и делает `cv.notify_all()`.
- Когда клиент вызывает `future.get()`, он захватывает этот же mutex и проверяет флаг. Если данные не готовы, он засыпает на `cv.wait()`.

> [!fail] Недостатки `std::future`
> 1. **Полное отсутствие композиции:**
> 	В стандартных `std::future` (вплоть до C++23) **нет** методов `.then()`, `when_all` или `when_any`! Из-за этого нельзя построить неблокирующий асинхронный pipeline.
>     
> 2. **Тяжеловесность:**
> 	Каждая связка promise/future вынуждена аллоцировать Shared State в куче и таскать за собой тяжёлые системные mutexes и [[Condvar|condvars]] ОС. Если в системе крутятся миллионы мелких асинхронных тасок, аллокации и блокировки на системных примитивах создают огромную нагрузку.
>     
> 3. **Одноразовость:**
>     Можно вызвать метод `.get()` на объекте `std::future` только **один раз**. Метод `.get()` перемещает *(move-семантика)* значение из Shared State, после чего состояние разрушается, а повторный вызов приводит к исключению.
>     
> 4. **Блокирующий деструктор `std::async`:**
>     Если `std::future` получена из вызова `std::async(std::launch::async, ...\)`, то её деструктор обладает специфическим поведением: он блокирует текущий поток до завершения работы асинхронной задачи, если результат ещё не забран. Это часто ломает ожидаемую асинхронность и приводит к скрытым дедлокам или просадкам производительности.

## Проблема Callback Hell

==Опр.== **Callback** - функция, которая передаётся в другую функцию и вызывается автоматически после завершения какого-то определённого действия.

==Опр.== **Callback Hell** - проблема глубокой вложенности кода при выполнении последовательности зависимых асинхронных операций.

```c++
// Пример Callback Hell
WhenRpcCallFinished(input, [](Output output) {
    ReadFileAsynchronously(output.filePath, [](Buffer buffer) {
        WriteToDatabaseAsynchronously(buffer, [](Status status) {
            std::cout << "Успешно завершено!" << std::endl;
        });
    });
});
```

### Ключевые недостатки Callback Hell

1. **Инверсия управления и структуры кода:** Весь код оказывается «вывернутым наизнанку». Вместо линейной последовательности действий код растёт вширь, напоминая пирамиду. Это критически ухудшает читаемость.
2. **Сложность обработки ошибок:** Классические исключения (`try-catch`) перестают работать, поскольку callbacks выполняются в других потоках выполнения. Разработчик вынужден вручную пробрасывать объекты ошибок через параметры каждого callback на каждом уровне вложенности.
3. **Невозможность нормальной декомпозиции:** Логику трудно разбить на отдельные переиспользуемые функции.

## Future Combining

==Опр.== **Continuation** - это возможность «подвесить» callback прямо на объект `Future` с помощью метода `.then()` или `.thenValue()`. Функция-callback автоматически выполнится, как только `Future` перейдет в состояние готовности.

==Опр.== **Chaining** - последовательная цепочка вызовов Continuation без вложенности. 

```c++
// Решение Callback Hell с помощью Chaining
Future<Output> f1 = SomeRpcCall(input);
Future<Status> f2 = std::move(f1)
    .thenValue([](Output output) {
        return ReadFileAsynchronously(output.filePath); // возвращает Future<Buffer>
    })
    .thenValue([](Buffer buffer) {
        return WriteToDatabaseAsynchronously(buffer);   // возвращает Future<Status>
    });
```

==Опр.== **Combining** - комбинирование или параллельная композиция нескольких задач.

#### Примеры Combining

1. **`when_all` (или `collect_all`):**
	 - Возвращает одну агрегированную `Future`, которая перейдет в состояние готовности **только тогда, когда ВСЕ** входящие future завершатся (неважно - успешно или с ошибкой).

2. **`when_any` (или `collect_any`):**
    - Она становится готовой, как только завершится **хотя бы одна (самая первая)** из входящих задач.

## Executors

==Опр.== **Executor** - это сущность, которая берёт на себя управление тем, ***где и как*** этот код будет физически выполнен *(на каком ядре, в каком потоке или контексте)*.

Без executor использование цепочек `.then()` становится опасным: callback, скорее всего, выполнится в контексте того потока, который вызвал `promise.set_value()`. Это может нарушить потокобезопасность или заблокировать критически важный поток.

Базовый интерфейс обычно представляет собой структуру с чистой виртуальной функцией `Execute`:
```c++
struct IExecutor {
    using Task = std::function<void()>;
    virtual void Execute(Task task) = 0; // Принимает задачу и планирует её выполнение
    virtual ~IExecutor() = default;
};
```

### Основные типы executors

1. **`ThreadedExecutor`:**
    При каждом вызове `Execute` лениво создаёт новый физический поток ОС (`std::thread`\), выполняет в нём задачу и завершает поток. Обладает колоссальной проблемой на частое создание потоков, поэтому используется редко.
    
2. **`ThreadPoolExecutor`:**
	При старте создаёт фиксированное или динамическое количество постоянно живущих рабочих потоков. Внутри содержит потокобезопасную (блокирующую) очередь задач. Потоки-работники забирают задачи из очереди и выполняют их, исключая накладные расходы на постоянное создание новых потоков.
    
3. **`SerializedExecutor`:**
    Специальный декоратор над другим executor, который гарантирует, что отправленные в него задачи будут выполняться **строго последовательно**, одна за другой.
    - _Зачем нужен:_ Позволяет организовать работу с разделяемым состоянием какого-то объекта без явного использования мьютексов *(нет параллельного доступа $\to$ нет Data Race)*.

Связующим звеном между продвинутыми future и executors является метод `.via(IExecutor*)`. Он осуществляет переключение контекста выполнения.
```c++
folly::ThreadPoolExecutor pool(4);
Promise<int> p;

// Привязываем будущую цепочку выполнения к пулу потоков pool
Future<int> f = p.getSemiFuture().via(&pool); 

auto f2 = std::move(f).thenValue([](int result) {
    // Этот код гарантированно выполнится внутри одного из потоков pool, 
    // независимо от того, какой поток вызовет p.setValue()!
    return result * 2;
});
```
